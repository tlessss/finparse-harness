# FinParseAI 全自主解析 2025 年报 — 计划表

> 目标：让系统**纯自主**解析 2025 年全部财报，正确率优先，垃圾数据无意义。
> 创建日期：2026-06-22 ｜ 状态图例：⬜ 未开始 / 🟡 进行中 / ✅ 完成

## 0. 目标与边界（已拍板）

| 决策项 | 结论 |
|---|---|
| **完成标准** | 净通过率 **≥95% 全自主**；剩余 ≤5% 长尾（图片型/加密/奇葩版式）归入死信队列，**允许人工介入** |
| **校验信任度** | 校验器（LLM+向量）作主裁判，但**关键字段（营收/研发）必须保留勾稽硬规则**，极少数边界可人工介入 |
| **第一优先级** | **正确率 > 一切**。先冲泛化与正确率，再做规模化；绝不批量产出错误数据 |

### 基线（2026-06-22 实测）
- 2025 年待解析 PDF：**1251 份**（`../book-agent/output/pdf_cache`）
- 已处理样本 18 份：净通过 4 / 需复核 14 → **实测净通过率 ≈22%**
- 三处空转待修：`optimizer.apply()` 不改规则、`auto_iterate._try_quick_fixes()` 恒返回 False、迭代仅用同规则重跑
- 页码/列索引硬编码（仅适配多氟多）

### 通往 95% 的 5 个必备能力
```
[泛化] 每家都能定位到表 → [自主修复] 错了能自己改对 → [自判] 不靠人也知道对没对
→ [规模] 1251 份跑得动、断了能续 → [收敛] 难例越来越少直到 ≤5%
```

### 已修复的真实 bug
1. **营收 `合计` 行被当分项**（占比之和翻倍，如 000002 分地区 200%）→ 加 `is_total_row()` 统一剔除。
2. **doc_type / 银行解析器误判**：`吸收存款/发放贷款` 出现在任何公司现金流量表，会把普通公司误判成银行 → 改用银行特有的 `非利息净收入` 计数判据。

### 关键测量（同一批 60 家，可复现对比）
| 指标 | 修复前 | Phase1 后 |
|---|---|---|
| 硬规则清洁率 | 41.7% | 43.3% |
| 净通过率(6/6+清洁) | 3.3% | 3.3% |
| 主要红线 | revenue:ratio_sum(30), employees:count_sum(12), rnd(4) | ratio_sum(29), count_sum(12), rnd(4) |

### ⚠️ 重要发现（决定 Phase 1/2 走向）
1. **`合计` 修复只是小赢**：营收 `ratio_sum` 红线的主因不是合计行，而是**漏行（占比之和 57~87%，表格抽取漏掉部分产品/地区行）**。这是 Phase 1.2「表头驱动列识别 + 跨表/多行合并」要解决的硬骨头，需结合人工 review 逐版式调，不宜无监督盲改。
2. **银行营收过度计数（333%）**走的是 default 解析器路径，需修银行解析器分支。
3. **架构级阻塞（影响 Phase 2）**：当前 content-based 解析器**完全不读 YAML 规则**（`revenue/default.py` 等的 `parse()` 从不用 `self.rule`）。所以优化 Agent「改 YAML」对解析行为**零影响**——这正是旧迭代闭环空转的根因。已为营收解析器接入第一个可调旋钮 `revenue_section.extra_exclude_names`（沙箱可调参验证），但**Phase 2 要真正生效，必须先把解析器改造成 rule-driven**（把关键词/排除项/阈值外置到规则）。这是 Phase 2 的前置任务，已记入下方。
