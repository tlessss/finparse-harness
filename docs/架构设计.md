### 2.2 数据流（输入/输出）

```
输入:
  PDF 文件路径（来自缓存或上传）
  + stock_code (如 "002407")
  + report_year (如 2025)

引擎内部:
  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
  │ pdfplumber   │───▶│ 解析器逻辑    │───▶│ JSON 组装    │
  │ 提取表格     │    │ 6个字段各自   │    │ 统一结构输出  │
  │ (替代Camelot) │    │ 列索引解析   │    │              │
  └──────────────┘    └──────────────┘    └──────────────┘

输出（最终 JSON）:
  {
    stock_code, company_name, report_year,
    pdf_file, parse_time, parse_duration_sec,
    revenue_breakdown: { segments[], industries[], regions[] },
    rnd_info: { rnd_detail[], total_this, total_last },
    employees: { total, parent, composition[], education[] },
    cost_breakdown: [{ industry, item, amount_wan, ratio_pct }],
    top_clients: { items[], total_ratio_pct },
    top_suppliers: { items[], total_ratio_pct },
    parse_flags: { rev: "ok", rnd: "ok", emp: "ok", ... },
    field_count: 6,
    db_write: "success"
  }
```

## 六、当前焦点：跨版式自适应

当前 6 个解析器在多氟多上可以完整跑通，但其他公司年报版式不同时会取不到数据。核心改进方向：

### 6.1 问题

```
当前:
  YAML 中 hardcode pages: "24-26"（仅适用于多氟多）
  → 澜起科技年报的营收结构表在第几页？不确定

目标:
  自适应搜索：用关键词 "主营业务分产品" 或 "营业收入构成"
  在全文本中搜索 → 找到关键词所在页码 → 提取该页表格
```

### 6.2 改进方案

1. 在 YAML 的每个 section 中保留 `keywords`（已有），新增 `auto_search: true/false`
2. 解析器启动时先用 PyMuPDF 全文搜索关键词找到页码
3. 如果用关键词找页码失败，fallback 到配置的 `pages` 默认值
4. 首次解析成功后，将实际页码回写到 YAML（可选）
