# 控制台测试指南（照着点就能验）

> 目的：一步步验证"人在回路控制台"是否就绪——既能看可信数据(绿)、也能看待办(红橙黄)，能审核/认证/控批量。
> 前置：后端 `:8200`、前端 `:3000` 都在跑。控制台地址：**http://localhost:3000/console**

---

## 0. 启动检查（先确认两个服务都活着）

```bash
curl -s -o /dev/null -w "后端 %{http_code}\n" http://localhost:8200/health     # 期望 200
curl -s -o /dev/null -w "前端 %{http_code}\n" http://localhost:3000/console    # 期望 200
```
- 后端没起：`cd ~/formal/FinParseAI && python3 -m src.api`
- 前端没起：`cd ~/formal/FinParseAI/frontend && npm run dev`

> 页面右上角应显示 **"实时"** 角标（连上后端）。若显示 **"示例"**，说明没连上后端 → 回去启动 :8200。

---

## 1. 准备数据（先扫几份，页面才有东西看）

```bash
curl -s -X POST http://localhost:8200/triage/scan \
  -H "Content-Type: application/json" \
  -d '{"codes":["000425","300005","300009"],"year":2025}'
```
扫完应有：**已核验 3 / 解析出 11 / 待办 15**（3 份 × 6 字段 = 18）。

---

## 2. 分诊覆盖台（重点：先看"可信"，再看"问题"）

打开 `/console`，进**分诊台**：

| # | 操作 | 期望（通过标准） |
|---|---|---|
| 2.1 | 看顶部 4 个大卡 | 🟢已核验 **3 (17%)** / 📊解析出数据 **11 (61%)** / 🔴待办 **15** / 净通过率 **17%** |
| 2.2 | 点 **🟢可信清单** 按钮 | 表格切到 **3 条绿记录**：000425营收、300005研发、300009成本；状态列显示"🟢可信"、置信度绿点 high |
| 2.3 | 点 **待办分级** 里的"需写解析器(红)" | 表格只剩 needs_write 那几条 |
| 2.4 | 点"待核验(黄)" | 只剩 unverified（多为客户/供应商/员工——无 DB 锚） |
| 2.5 | 点"低置信(橙)" | 只剩 1 条：300005 成本（按产品成本，锚对不上）|
| 2.6 | 看 4 级配色 | 绿=可信 / 黄=待核验 / 橙=低置信 / 红=需写，颜色分明 |

✅ **过关标准**：你第一眼看到的是"已核验/解析出"两个绿蓝大数字，而不是一堆红——有安全感。

---

## 3. 审核台（看对不对 + LLM 裁判 + 认证）

在分诊台任意一条点 **「审核」**：

| # | 操作 | 期望 |
|---|---|---|
| 3.1 | 进审核台 | 左侧解析结果各字段；右侧 PDF 渲染页 |
| 3.2 | 看溯源红框 | 解析出的值在 PDF 原文上有红框对照（点字段高亮）|
| 3.3 | 看置信度徽章 | 每字段 high(绿)/low(橙)/unknown(灰)；high 的显示 DB 锚值 |
| 3.4 | 触发 **LLM 裁判** | 等 ~10-20s → 出 verdict(ok/suspicious) + issues(现值→正确值+错误类型+源文依据) |
| 3.5 | 拿一条正确的（如 300005 研发）跑裁判 | 期望 **ok**（"与源文一致"）|

> 想看 LLM 抓错：审核台若能手动改值，把某金额改小 10 倍再跑裁判 → 期望 **suspicious + unit_error**。

**认证写回**（产）：人确认无误后
- 存 golden（`/review/golden`）→ 认证解析器（`/review/certify`，服务端重验 exact）→ 该字段从待办**自动销账**、下次同版式自动路由。

---

## 4. 批量控制：任务列表 → 开始解析（★ 你要的入口，从这测起）

进**批量控制**页：

| # | 操作 | 期望（通过标准） |
|---|---|---|
| 4.1 | 看「任务列表」 | 一排可解析报告（勾选框），**默认勾了前 10 份** |
| 4.2 | 用默认 10 份（或自己勾几份）→ 点 **▶ 开始解析（N 份）** | 状态变"● 运行中"，进度条开始动，"最近处理"逐条刷新（每 2s 自动轮询）|
| 4.3 | 看进度数字 | done/total 往上走；字段有数据(绿)/需写(红)/待核验(黄)/异常 实时更新 |
| 4.4 | 点"暂停"→"继续"→"停止" | **停止后 done 不再涨、状态变"已停止"**（这条之前修过 race，重点验）|
| 4.5 | 跑完回**分诊台** | 覆盖台账多了刚解析的报告，verified%/待办数字变化 |

> ⚠️ **别点"全选"跑 1249 份**；测试就默认 10 份。
> 后端单跑：`python scripts/run_batch.py 000425 300005 300009`（无参数不会跑全量，会提示用法）。

---

## 5. 复核闭环（#1 筛 → #2 判 → 自动收口）

```bash
# 对队列里"低置信"跑 LLM 复核：对的销账、错的改标
curl -s -X POST http://localhost:8200/triage/review \
  -H "Content-Type: application/json" -d '{"reason":"low_confidence","limit":5}'
```
- 期望：300005 成本(低置信)被 #2 看了真实原表 → 判 **ok** → **自动移出待办**。
- 再看分诊台：低置信那条没了，已核验/解析数可能上升。

## 9. 验收标准（这些都 ✅ 就算控制台就绪）
- [ ] 顶部能看到 **verified%/parsed%** 安全感数字（不只是问题列表）。
- [ ] 🟢可信清单点得开、能看到具体绿记录。
- [ ] 审核台：解析值 + PDF 红框 + 置信度 + LLM 裁判 都能出。
- [ ] 认证一条 → 它从待办销账。
- [ ] 批量控制：起/停/进度可用（停止真能停）。
- [ ] 后端关掉时页面优雅降级（示例/报错态，不白屏）。
