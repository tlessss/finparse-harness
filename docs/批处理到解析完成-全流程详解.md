# 从批处理到解析完成 · 全流程详解（带 mock 数据）

> 贯穿示例：**002254 泰和新材 2025 年报**（两年并排占比构成表，营业收入 ≈ 35.95 亿）。
> 每个关键节点给出**真实形状的 mock 数据**，对照代码看。涉及文件：
> `src/batch_runner.py`、`src/engine_orchestrator.py`、`src/eval/table_cache.py`、
> `src/parsers/infra/table_scanner.py`、`src/parsers/revenue/default.py`、
> `src/parsers/revenue_router.py`、`src/eval/anchors.py`、`src/eval/triage_queue.py`。

---

## 0. 全景：一份报告要穿过的关

```
点击开始解析
   │  POST /batch/start {codes, year, heal, step}
   ▼
run_batch 线程  ── 逐 code 循环 ──┐
   │                              │
   ▼                              │
①取表 get_tables ───────────────┐│
   │  (缓存命中→秒回；否则 scan_pdf 全章节扫)
   ▼                            ││
②引擎 engine.run 逐字段：       ││
   ├ 路由 route_field(指纹)     ││  命中认证解析器 → 用它(零LLM)
   │   └ 没命中 → 冷启动解析器  ││
   ├ 选表 filter_by_signature  ││  caption+章节+内容打分
   ├ 解析 parser.parse         ││  → 结构化分项
   ▼                            ││
③判定 field_plausibility ──────┘│  分项和 对 DB锚 → high/low/unknown
   ▼                             │
④质检关(引擎signals) ───────────┘  trust? → 写库 / 不写+送审
   ▼
⑤台账 triage_report  →  ok(绿)/needs_write(红)/low(橙)/unverified(黄)
   ▼
⑥(若 heal) 自愈：抽golden→写解析器→认证   （另见《自愈流程详解》）
   ▼
⑦进度回传 batch_state.json  →  前端轮询 /batch/progress
```

## 1. 批处理启动

**前端**：批量控制页勾选报告 → 点"开始解析" → `POST /batch/start`。

**请求体（mock）**：
```json
{ "codes": ["002254", "000425", "000333"], "year": 2025, "heal": false, "step": false }
```

**后端**（`src/api.py` `batch_start`）：拒绝重复运行，起后台线程：
```python
threading.Thread(target=run_batch,
                 args=(req.codes, req.year, req.db_write, req.heal, req.step)).start()
```

**初始 `goldset/batch_state.json`（mock）**：
```json
{
  "running": true, "total": 3, "done": 0, "skipped": 0, "errors": 0,
  "fields_with_data": 0, "by_reason": {},
  "started_at": "2026-06-29 15:10:00",
  "recent": [], "current": null, "stage": null
}
```

> `run_batch` 每份报告前都 `_read().get("stopped")` 查停止标志；这就是"停止"如何生效。

## 2. ① 取表 get_tables（`src/eval/table_cache.py`）

```python
pre = get_tables("002254", 2025)   # 缓存命中→直接读json；否则 scan_pdf 全章节扫+落盘
```

**scan_pdf 做什么**：
- `detect_page_context`：优先用 PDF 大纲 `get_toc()` 给每页打章节标签（`management`/`gov`/`fuzhu`/`other`）。
- 章节驱动选页：有书签 → 只扫 `management+fuzhu+gov`；无书签 → 全扫 16~min(末页,350)。
- 每页 `find_tables()` 抽表，并抽**上文标题 caption**（表正上方同页文本）。

**一张表的形状（mock，002254 营收构成表所在 page22）**：
```json
{
  "page": 22,
  "caption": "1、收入与成本 （1） 营业收入构成 单位：元",
  "section": "management",
  "table": [
    ["项目", "", "", "2025年", "", "", "", "占营业收入比重", "", "2024年", "...同比"],
    ["分行业", "", "", "", "", "", "", "", "", "", ""],
    ["化学纤维产品", "", "", "3,536,510,944.62", "", "", "", "98.37%", "", "3,869,600,726.55", "..."],
    ["其他", "", "", "58,665,113.86", "", "", "", "1.63%", "", "...", "..."],
    ["分产品", "", "", "", "", "", "", "", "", "", ""],
    ["先进纺织品", "", "", "1,333,630,882.01", "", "", "", "37.10%", "", "...", "..."]
  ],
  "table_bbox": [51.3, 410.2, 540.0, 690.5],
  "cell_bbox": [[...每格(x0,y0,x1,y1)或null...]]
}
```

> 注意三个为选表服务的字段：**caption**（标题，最强信号）、**section**（章节，弱先验）、**table**（内容）。
> 两年并排：col3=2025金额、col7=2025占比、col9=2024金额——**列是错位的**（坑）。

## 3. ② 引擎逐字段（`src/engine_orchestrator.py` `run`）

引擎对 6 个字段各跑一遍"**先路由、没命中再冷启动**"。以**营收**为例。

### 3a. 路由 route_field（`src/parsers/revenue_router.py`）—— 选择即验证
```python
rev_result = self._route_field(REVENUE, "002254", 2025, all_tables)  # 命中认证解析器?
if rev_result is None:
    rev_result = rev_parser.parse(pdf, pre_scan=all_tables)          # 没命中→冷启动通用器
```
- 算这份报告的**版式指纹** `fingerprint_of("002254",2025)`（mock：`"fp_a1b2c3"`）。
- 查认证目录里**指纹命中**的候选解析器；跑它们、对硬规则判（选择即验证）。
- 002254 已认证专用解析器 → **routed**，用 `rev_002254_v1`，零 LLM。

**路由结果（mock）**：
```json
{ "status": "routed", "parser_key": "002254-泰和新材-两年并排营收",
  "result": { "...见下..." } }
```

### 3b. 选表 filter_by_signature（冷启动/找候选时；`table_scanner.py`）
给每张候选表打分：**caption 命中 +40 → 章节弱先验 ±20/8 → 表体 must_have +25/个 → 占比列 +20 → 金额列 +15 → 同比列误当占比 −50**。

**打分结果（mock）**：
```json
[
  { "page": 22, "score": 205 },   // caption"营业收入构成"(+40)+章节+占比列+...  ← 选中
  { "page": 23, "score": 137 },   // "占营收10%以上"辅助表
  { "page": 60, "score":  70 }    // 利润表(只有营业收入,无分项)
]
```
> caption 把**主营收构成表(22)** 和 **10%以上辅助表(23)** 精准分开——这是 caption 主信号的价值。

### 3c. 解析 parser.parse（专用器 `rev_002254_v1` 或通用 `revenue/default.py`）
专用器的逻辑：占比列取**最左 0~100% 列**（=本期2025）、金额列取占比左侧、按 `分行业/分产品/分地区/分销售模式` 切桶、跳合计行。

**解析输出 revenue_breakdown（mock）**：
```json
{
  "industries": [
    { "name": "化学纤维产品", "revenue_yuan": 3536510944.62, "ratio_pct": 98.37 },
    { "name": "其他",         "revenue_yuan":   58665113.86, "ratio_pct": 1.63 }
  ],
  "segments": [
    { "name": "先进纺织品",                   "revenue_yuan": 1333630882.01, "ratio_pct": 37.10 },
    { "name": "安防与信息及新能源产业用产品", "revenue_yuan": 2214237622.86, "ratio_pct": 61.59 },
    { "name": "其他",                         "revenue_yuan":   47307553.61, "ratio_pct": 1.31 }
  ],
  "regions": [
    { "name": "国内地区", "revenue_yuan": 2811639466.39, "ratio_pct": 78.21 },
    { "name": "国外地区", "revenue_yuan":  783536592.09, "ratio_pct": 21.79 }
  ],
  "by_channel": [
    { "name": "订单销售", "revenue_yuan": 3595176058.48, "ratio_pct": 100.00 }
  ]
}
```
> 每个维度的金额之和 = **3,595,176,058**。记住这个数，下一关要拿它对锚。

## 4. ③ 判定 / 跨表锚验证（`anchors.py` + `revenue_router.py`）

### 4a. 取锚 get_anchors —— 独立权威真值
从 DB（`income_statement` JSON）拿这份报告的权威数：
```python
anchors = get_anchors("002254", 2025)
```
**锚（mock）**：
```json
{ "revenue": 3595176058.48, "cost": 2705331189.00, "rnd_expense": 215430000.00 }
```

### 4b. field_plausibility —— 把"对不对"变成算术对账
```python
sig = field_plausibility(REVENUE, rev_result, anchors)
```
逻辑（A 类 = 锚为主、占比为辅）：
- 取各维度金额和的**最接近锚的那个** `_parsed_total` = 3,595,176,058。
- `anchored = |total − anchor| ≤ 3% × anchor` → True（3.595B vs 3.595B，几乎为0）。
- `clean = anchored OR 各维度占比和≈100`。
- `confidence = high(过锚) / low(有锚但对不上) / unknown(无锚)`。

**判定信号（mock，过锚）**：
```json
{ "clean": true, "anchored": true, "confidence": "high",
  "anchor": 3595176058.48, "parsed_total": 3595176058.48, "n_dims": 4, "rows": 8 }
```

**反例 mock（冷启动抓成 2024 数据）**：分项和=3,929,372,304（2024），对 2025 锚差 9% →
```json
{ "clean": false, "anchored": false, "confidence": "low", "anchor": 3595176058.48 }
```

## 5. ④ 质检关（`engine_orchestrator.py` 后处理）—— 决定写不写库

引擎对**每个字段**统一过锚，决定可信度与是否写库：
```python
_trust = _routed or (_sig["confidence"] != "low")   # 认证路由背书 OR 锚没判错
if not _trust: del db_fields[field]                  # 不可信 → 不写库
output["signals"][field] = {source, confidence, clean, anchored, written}
```

**引擎输出（mock，002254 营收）**：
```json
{
  "revenue_breakdown": { "...见上..." },
  "signals": {
    "revenue_breakdown": { "source": "routed", "confidence": "high",
                           "anchored": true, "clean": true, "written": true }
  },
  "parse_flags": { "revenue_breakdown": "ok" },
  "溯源": { "revenue_breakdown": { "industries[0].revenue_yuan": {"page":22,"bbox":[...]}, "..." } }
}
```

**对照 mock（冷启动+low → 挡在库外）**：
```json
{ "source": "cold_start", "confidence": "low", "anchored": false, "written": false }
```
> **铁律**：认证路由=背书写库；冷启动只有锚没判错才写；**锚判 low → 绝不写库**，标 `written:false` 送审。

## 6. ⑤ 台账 / 分诊（`src/eval/triage_queue.py` `triage_report`）

把每个字段的结局落进覆盖台账（按 code+year+field **去重 upsert**）：

**台账记录（mock）**：
```json
[
  { "code":"002254","year":2025,"field":"revenue_breakdown",
    "status":"ok","reason":"routed",
    "signal":{"confidence":"high","diff_pct":0.0}, "updated_at":"2026-06-29 15:10:42" },

  { "code":"002254","year":2025,"field":"cost_breakdown",
    "status":"open","reason":"needs_write",
    "signal":{"confidence":"unknown"}, "updated_at":"..." }
]
```

**状态 → 前端颜色**：
| status / reason | 含义 | 颜色 |
|---|---|---|
| ok / routed(high) | 锚验证过、可信 | 🟢 绿 |
| open / needs_write | 有数据但无认证解析器 | 🔴 红 |
| open / low_confidence | 锚对不上 | 🟠 橙 |
| open / unverified | 无锚、没法验 | 🟡 黄 |
| healed | 自愈写出并认证 | 🟢✨ |

## 7. ⑥ 自愈分支（仅 `heal=true`；详见《自愈流程详解》）

对 `needs_write/low_confidence` 且**有锚**的字段触发。串行：

1. **LLM 抽 golden（必须过锚）** → `llm_extract_golden`
   ```json
   // golden(mock,002254成本,过锚后)
   [ {"name":"原材料","amount_yuan":1623198713,"ratio_pct":60.0},
     {"name":"职工薪酬","amount_yuan":541066237,"ratio_pct":20.0}, "..." ]
   ```
2. **组 golden_entry**（必带 `_status`，否则 eval 越界）
   ```json
   { "stock_code":"002254","year":2025,"_status":"confirmed_auto",
     "cost_breakdown":[ "...golden..." ] }
   ```
3. **repair 三岔**（选择即验证选母本）→ mock：
   ```
   pick_mother → ("cst_300005_v1.py", 0.46, "300005成本") → 0.3≤0.46 → fork
   ```
4. **generate_parser 迭代**（不给 golden 真值，沙箱对 golden 打分，必须 exact）→ mock：
   ```
   [第1轮] 分=0.0  exact=False
   [第2轮] 分=0.53 exact=False
   [第3轮] 分=1.0  exact=True   → 🎓 认证
   ```
5. **认证 + 路由**：写 `cst_002254_2025.py`，登记指纹 → 下次同版式零 LLM。
   ```json
   // certified_parsers.json 新增(mock)
   { "key":"002254-2025-cost_breakdown-自愈", "path":"src/parsers/versions/cst_002254_2025.py",
     "field":"cost_breakdown", "fingerprints":["fp_a1b2c3"] }
   ```
   不到 exact → 删孤儿文件 + 转人工（`needs_human`）。

## 9. 一句话钉死全流程

> **点开始 → run_batch 逐份 → 取表(缓存/全章节扫,带caption+章节) → 引擎逐字段[路由命中认证器/否则冷启动 → caption+内容选表 → 解析出分项] → 拿分项和对 DB锚(把对错变算术) → 质检关按可信度决定写库/送审 → 落台账(绿/红/橙/黄) →（heal 时失败字段自愈：抽过锚golden→写解析器→认证）→ 进度回传前端。**

**两条贯穿主线**：
1. **可信度全程靠"跨表锚"**（分项和≈营业收入），不靠 LLM 拍脑袋——这是"正确率优先"的根。
2. **运行期零 LLM**：命中认证解析器就是跑确定性代码；LLM 只在构建期(自愈)写代码。
