# 新版解析技术路线（规范驱动重写）

> 目标：把当前"模糊关键词 + 统计认列"的启发式解析，重写为**规范驱动（spec-driven）+ 规则化（rule-driven）**的解析管线。
> 依据：`docs/规范要点-解析映射.md`（准则第2号 2025版提炼）。
> 衡量：所有改动都用 `scripts/baseline_run.py` / `autonomous_run.py` 在固定样本上 A/B 对比，硬规则清洁率提升才合并。
> 创建：2026-06-24

---

## 一、为什么要重写（现状三宗罪）

1. **找页**：模糊关键词高召回低精度 + `[:15]` 按页码盲截断 → 真表页可能进不了候选。
2. **认表**：只取"最高分 1 张表"，分不清 (A) 占比构成表 / (B) 毛利率表 → 挑错表（徐工 000425）。
3. **认列**：纯统计、不读表头 → 把毛利率当占比。
4. （根因）解析器**不读规则**，优化 Agent 无从调参。

全量 1249 份：硬规则清洁率仅 44%，死信 699 里 643 是营收。

## 二、设计原则

- **规范优先**：能从准则/规则确定的，绝不靠猜（章节锚点、表头白名单、维度、标准科目）。
- **正确率优先**：宁可置空，不可错填（如无"占比"列就置空，禁止拿毛利率顶替）。
- **规则驱动**：解析行为由 YAML 规则控制，解析器只执行；解锁优化 Agent 的可调旋钮。
- **可度量、可回滚**：每步与现版本在固定样本对比；sandbox 验证不退步才合并。
- **渐进式**：先攻最大瓶颈（营收），跑通再推广，不一次性大改。

## 三、新解析管线（每个字段）

```
scan_pdf(pdf)  ─ 已有：扫表 + 章节上下文标签（附注/MD&A/其它）
      │
      ▼  对每个字段，按其规则执行：
┌─────────────────────────────────────────────────────────────┐
│ ① SectionLocator  章节/小节锚定                                │
│    用 rule.sections 把候选表限定在对的章节（如 MD&A"收入与成本"）│
│      ↓                                                         │
│ ② TableClassifier 表语义分类                                   │
│    用 rule.table_types 给候选表打类型标签                       │
│    （revenue-A 占比构成表 / revenue-B 毛利率表 / cost-性质 …）  │
│      ↓                                                         │
│ ③ HeaderDrivenColumns 表头驱动认列                             │
│    用 rule.header_aliases 匹配表头文字 → 列映射                 │
│    （找不到"占营业收入比重"列 → ratio_col=None，占比置空）      │
│      ↓                                                         │
│ ④ RowExtractor 行抽取                                          │
│    按 rule.dimensions 切桶（行业/产品/地区/销售模式），跳合计行 │
│      ↓                                                         │
│ ⑤ FieldValidator 分表校验                                      │
│    按表类型选口径：A→占比和≈100；B→分项收入和≤营收合计          │
└─────────────────────────────────────────────────────────────┘
```

核心变化：**②③ 是全新的（语义认表 + 表头认列），替换现在的"统计法挑表/认列"。** ①④⑤ 在现有基础上规则化。

## 四、规则 Schema（rule-driven 的核心）

每个字段一份规则，外置到 `src/parser_rules/`，由解析器读取执行。营收示例：

```yaml
revenue_breakdown:
  sections:                      # ① 章节锚点（小节句式）
    anchors: ["管理层讨论与分析", "收入与成本", "主要经营情况"]
    section_tag: management      # 复用 table_scanner 的章节标签
  table_types:                   # ② 表语义分类
    composition:                 # (A) 占比构成表 —— 我们要的
      require_any_header: ["占营业收入比重", "营业收入比重", "占比"]
      validate: ratio_sum_100
    margin:                      # (B) 收入/成本/毛利率表 —— 避坑
      require_any_header: ["毛利率"]
      forbid_as_ratio: ["毛利率"]   # 禁止把毛利率列当占比
      validate: revenue_le_total
  header_aliases:                # ③ 表头驱动认列
    name:   ["项目", "分行业", "分产品", "分地区", "分销售模式"]
    revenue:["营业收入", "金额", "收入"]
    ratio:  ["占营业收入比重", "营业收入比重", "占比"]
    cost:   ["营业成本"]
    gross:  ["毛利率"]
  dimensions:                    # ④ 维度锚点（含新增"销售模式"）
    分行业: industries
    分产品: segments
    分地区: regions
    分销售模式: by_channel
  exclude_rows: ["合计", "小计", "总计", "其中"]
```

> 这份 schema 同时是 **Phase 2.0 优化 Agent 的可调旋钮面**：诊断后改的就是 `header_aliases` / `sections.anchors` / `exclude_rows`，再用 sandbox 验证。

## 五、四个核心新组件（代码落点）

| 组件 | 职责 | 落点 | 复用现有 |
|---|---|---|---|
| `RuleLoader` | 加载字段规则 YAML | `src/parser_engine/rule_loader.py`（已存在，接上） | ✅ |
| `SectionLocator` | 章节/小节锚定 → 候选页/表 | `src/parsers/infra/section_locator.py`（新） | 复用 `detect_page_context` |
| `TableClassifier` | 给表打语义类型标签 (A/B/…) | `src/parsers/infra/table_classifier.py`（新） | 复用 `filter_by_signature` 思路 |
| `HeaderDrivenColumns` | 表头文字 → 列映射 | `src/parsers/infra/header_columns.py`（新） | 替换 `detect_column_types` |
| `FieldValidator` | 分表类型校验 | 扩展 `src/validators/hard_rules.py` | ✅ |

> 注：`src/parsers/infra/` 已被用于 `layout_fingerprint`，新组件归入同一 infra 包。

### 关键组件：HeaderDrivenColumns（解决"认错列"）
```
输入：表格二维数组 + rule.header_aliases
1. 在前 1~3 行定位表头行（含最多别名命中的行）
2. 对每个目标列（name/revenue/ratio/cost/gross），
   在表头行里按别名匹配 → 锁定列索引
3. ratio 列必须由"占营业收入比重"类别名命中才成立；
   若只命中"毛利率"→ ratio_col=None（占比置空，禁止顶替）
输出：{name_col, revenue_col, ratio_col, cost_col, gross_col}
```
统计法只作 **fallback**（表头缺失时）。

## 六、各字段实现路线（按优先级）

1. **营收（最大瓶颈，先做）**：A/B 表分类 + 表头驱动认列 + 销售模式维度 + 分表校验。验收用徐工类 case。
2. **成本**：区分"按性质(原材料/人工/折旧/能源动力)"与"按行业/产品"两类表；性质表校验占比和≈100。
3. **研发**：分清"附注研发费用科目表"（现有）与准则要求的"研发投入总额/占比/资本化"（新增字段）。
4. **员工**：用准则标准类别白名单（生产/销售/技术/财务/行政人员）；校验三和相等（已有）。
5. **客户/供应商**：以"4 个汇总比例"为主目标；明细名单缺失不算漏（按规范"鼓励"非强制）。

## 七、与现有代码的关系 / 迁移与回滚

- **不动现有 content-based 解析器**：作为 fallback 保留。
- **新建 rule-driven 解析路径**，按字段逐个切换；selector 可按规则/指纹选择走新路还是旧路。
- **A/B 对比护栏**：同一固定样本（如 `autonomous_run --limit 60`），新旧各跑一遍，清洁率/字段覆盖对比；**仅在不退步且有提升时合并**。
- **sandbox 把关**：规则改动经 `src/agents/sandbox.py` 验证（硬规则严格变好才接受）。

## 八、分阶段里程碑

| 里程碑 | 内容 | 验收 |
|---|---|---|
| **M1 地基** ✅ | RuleLoader + 营收规则 schema + HeaderDrivenColumns + 接入营收解析器 | ✅ 徐工 000425：毛利率不再当占比(占比置空、虚假64%红线消失)；多氟多仍100%；22/22 单测过 |
| **M2 营收重写** ✅ | A/B 表择优(优先占比表) + 销售模式维度 + by_channel 校验 | ✅ 固定60样本清洁率 **43.3%→68.3%(+25pt)**，15家变好，**0回归**；25/25单测 |
| **M3 全量验证** | 全量 1249 A/B 对比 | 营收死信占比明显下降，无整体退步 |
| **M4 推广** | cost / rnd / employee / clients 迁到新管线 | 各字段清洁率↑ |
| **M5 接优化 Agent** | 优化 Agent 调规则 schema + sandbox | 死信自动修复成功率 ≥50%，不把对的改坏 |

## 九、风险与对策

| 风险 | 对策 |
|---|---|
| 表头驱动在无框线/合并单元格表上仍认不到表头 | 保留统计法 fallback；必要时引入更强表格引擎（Camelot/MinerU）做对照 |
| 规则覆盖不全（新措辞） | header_aliases 可由优化 Agent 增量补充（Phase 2.0） |
| 新路径整体退步 | A/B 护栏 + sandbox，不退步才合并；可按字段灰度 |
| "销售模式"等新维度引入噪声 | 先做成可开关；用硬规则验证占比和 |
