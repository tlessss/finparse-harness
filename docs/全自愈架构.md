# 全自愈架构

> 目标:把营收/成本等字段的自主解析率从当前 ~62% 推向 95%,**正确率优先**——宁可不入库,绝不入错。
> 核心思想:让**每一类失败都有自动恢复路径**;系统自主地要么"入对的数据",要么"正确判定无数据/域外",人工只兜真正修不了的死角。

本文档钉住目标架构与落地顺序,逐步实现时对照更新。

---

## 一、原则

1. **正确率优先,双闸兜底**:任何自愈产出都必须过**两道闸**才入库——
   - **过锚闸** `field_plausibility`:每个解析维度合计 ≈ 营业收入锚(±3%)。
   - **复核闸** `verify_field`:LLM 逐项核 + 体检(选对表、够全)。
   正因为闸严,才敢堆很多 healer 而不怕入错(徐工"占10%以上"坑表就是被闸挡下的)。**闸是保护,不是障碍**——所以"单维不齐"这类绝不能靠松闸放行。
2. **终态诚实**:修不了的要**明确标注病因**交人工,不假入库;数据本就不存在的(真无表、金融股无营收构成)→ 正确判"无/域外"也是一种自愈。
3. **越跑越强(固化)**:每次成功自愈产出**可复用产物**(规则版本 / 认证解析器),下次同版式直接 routed、不再发 LLM。从"每次现修"变成"修一次永久会"。

---

## 二、现状:自愈是"固定级联",不是"按病下药"

当前 `pipeline.run_field` 的非绿灯路径是一条写死的链(`src/pipeline.py`):

```
route_field?               # ① 命中认证解析器 → routed 绿灯
  否 → _parse_versioned    # ② base → 规则版本池 → 跨页拼接(_stitch_cold)
        过锚 → _green_llm   #   复核 → pass 入库 / wrong_table → 选表自愈
        非绿灯 → _nongreen_llm
              → _heal_and_verify(选表自愈)
                    no_pick   → no_such_table(人工)
                    still_bad → _rule_heal_and_verify(L2改规则)
                                修不了 → judge_diagnose → 人工
```

**问题**:所有失败都挤在"选表自愈 → L2 → 诊断 → 人工"这一条上。
- "全空"(选表/认列全废)明明该被选表自愈/L2 救,却常走到诊断交人工;
- "无锚/抽表漏行/金融股"根本**没有对应 healer**;
- `judge_diagnose` 只会"交人工",不会"派活"。

已经具备但**触达不足**的能力:复核(`verify_field`)、选表自愈(`select_table_agent`)、L2改规则(`rule_heal_agent`)、跨页拼接(`table_recall.following_tables/merge_tables` + `_stitch_for_anchor`)、规则版本池(`rule_versions`)、选表避坑(`_is_ten_pct_trap`)。

## 三、目标架构:两层编排(干活的 + 掌舵的)

自愈分**两层**,各司其职:

- **Tier 1 · 确定性路由器**:处理 ~90% 清晰的失败——快、便宜、可复现。诊断明确就按类派 healer,走双闸入库/固化。**它扛主流量。**
- **Tier 2 · 超级管家 agent(目标感知)**:只在"难的尾巴"和"元决策"时出场——诊断歧义、healer 链跑完仍不过(交人工之前最后想想)、以及批量复盘驱动路线图。**它掌舵,不抢闸的活。**

类比:自动化 runbook(Tier 1)处理已知故障,资深值班工程师(Tier 2)处理怪的、并改进 runbook 本身。

### Tier 1:诊断驱动的迭代闭环

```
过锚失败
 → ① 诊断归类(是哪种病)         diagnose()
 → ② 路由到 healer 链(可升级)    route(diag) -> [healer1, healer2, ...]
 → ③ 逐个 healer 尝试            每个都【过锚闸 + 复核闸】gated
       过双闸 → 入库 + 固化(产物存起来,下次直接命中)
       不过   → 升级到链里下一个更重的 healer
 → ④ 整条链都不行 → 明确标注病因交人工(不假入库)
```

伪代码(拟落在 `src/agents/heal_orchestrator.py`):

```python
def heal_orchestrate(code, year, field) -> dict:
    diag = diagnose(code, year, field)          # 复用 _classify_failure + 必要时 LLM
    for healer in route(diag):                  # 升级链
        r = healer(code, year, field, diag)     # healer 内部已过"过锚闸"
        if r and r.get("anchored"):
            v = verify_trusted(code, year, field, r)   # 复核闸(信任源文)
            if v.get("verdict") == "pass":
                commit(code, year, field, r)
                consolidate(diag, healer, r)     # 固化:规则版本 / 认证解析器
                return {"outcome": "committed", "healed_by": healer.__name__}
    return {"outcome": "needs_human", "reason": diag.reason}   # 终态诚实
```

### 诊断器可直接复用已有的确定性归类

`pipeline._classify_failure`(失败分析页那个)已经能从"占锚形状 + 标题 + 表头 + 复核疑点"判出 10 类。把它从**分析器**升级成**路由器**即可,无需新造。歧义时再叠一层 LLM 诊断(`judge_diagnose`,但其 `next_action` 要真正映射到 healer 并被执行,而不是一律交人工)。

### Tier 2:超级管家 agent(目标感知的监督者)

一个**知道最终目标**的高层 agent。它**不是** per-report 路由器(那会又贵又不可复现,还会抢"双闸"保正确率的活),而是**难尾巴 + 元决策 + 治理**三合一。

**触发时机(Tier 1 兜不住时才叫它):**
1. 诊断**歧义 / 不属于已知 10 类**(路由器不知道派谁);
2. healer 链**跑完仍不过**——交人工之前,让它最后想想有没有创造性组合;
3. **元决策**:该不该判域外、该不该为这个版式固化一个永久解析器;
4. **批量复盘**(离线):一整批跑完后复盘失败,驱动路线图。

**它知道什么(输入):**
- **目标**:全部 2025 年报、自主率 95%、**正确率优先**、每份要么入对数据要么正确判"无/域外",人工只兜死角;
- **上下文**:当前成功率、失败分布(`failure_analysis`)、有哪些 healer、**这份已经试过什么**、它的全部表 + 锚 + 候选诊断;
- **历史**:本报告的自愈尝试轨迹(哪些 healer 试了、为何不过锚/不过复核)。

**它能拍板什么(职权):**
1. **编排策略**——不按固定链,自己决定先试哪个 healer、怎么组合(如"这份像跨页也像选错表,先重选再拼");
2. **判域外** `out_of_scope`——如金融股营收构成不适用,直接标,不浪费 healer;
3. **带病因交人工**——即便某 heal 过了闸,它不放心也可判"证据不足→人工",并写清原因(正确率优先);
4. **决定固化**——值不值得为这个版式建永久认证解析器(经验库)。

**治理角色(最值钱的一环):** 批量跑完后让管家复盘整批——
> "现在 62%,最大缺口是'全空 8 + 单维不齐 10',其中 X 家同一版式 → 建议建 healer/规则 Y。"

它不只修单份,而是**盯着 95% 的目标,指出下一步该造什么**。等于把路线图里"人来定优先级"那步,交给一个懂全局的 agent 持续做——与"越跑越强/固化"支柱同一条线。

**底线不变:管家提议,闸 enforce。** 它选/改出来的表仍要过**过锚 + 复核**才入库(徐工"10%以上"坑表就是被闸挡的)。所以加管家**扩大的是自动处理范围与决策智慧,不削弱正确率**。

落点:`src/agents/steward_agent.py`(新)。Tier 1 的 `heal_orchestrate` 在"链尽/歧义"处回调它;批量复盘用一个离线入口(读 `failure_analysis` + 样本链路)。

**观测前置**:管家读「这份已经试过什么」依赖**案件事件时间线**(`process_events` + 前端 Case Timeline),见 [`链路观测-案件时间线.md`](链路观测-案件时间线.md)。看不清链路就无法调 healer、也无法让管家有输入。

## 五、要新建的 healer(详细)

### 5.1 取锚自愈 + 金融股域外(治"无锚")

- **取锚自愈**:锚缺失时,从**合并利润表**里抽"营业收入"一行当锚(利润表几乎总能稳定抽到),再走正常过锚流程。落点:`src/eval/anchors.py` 加一个 `derive_anchor_from_income_statement(code, year)` 兜底,`field_plausibility` 无 DB 锚时调用。
- **金融股域外**:按股票名称/行业识别银行/证券/信托/保险(如名称含"银行/证券/信托",或行业代码属金融)→ 营收构成**不适用**,标 `out_of_scope`,**移出成功率分母**(现在 `failure_analysis` 已把"无锚"剔出分母,这里把它正名为"金融域外")。落点:`field_spec` 或一个 `_is_financial(code)` 判定。
- 收益:直接砍掉"无锚"整类。**最省事,先做。**

### 5.2 选表自愈多轮 + L2 迭代(治"选错表 / 全空 / 取错列")

- **选表多轮**:`select_table_llm` 现在单轮认不出就停。改成**扫 top-N 候选**,逐个 forced 重解 + 过锚,取第一个过锚的;都不过再退。避坑(`_is_ten_pct_trap`)已在确定性选表生效,LLM 选表也要带上。
- **L2 迭代**:`rule_heal` 提一版 delta 不过锚,允许**再提一版**(把上版为何不过锚回喂给 LLM),最多 K 轮。
- **认列扩展**:`rule_heal.yaml` prompt 补"单位 override(万元/千元)"和"名称别名/清洗",覆盖 amount_error / name_error。

### 5.3 抽表自愈(L3 重抽,治"严重缺 / 中度缺 / 真漏行")

- 真表选对但 pdfplumber 抽漏行(如盈方微把表抽碎)→ 现有规则/选表都够不着,必须**回到抽表层**:
  - 对目标页**换 pdfplumber 参数重抽**(不同 line/text 策略、放宽/收紧 snap);
  - 仍不行则**渲染该页区域为图 → LLM 抽行**(视觉兜底);
  - 产出的表走 forced 重解 → **过锚闸兜底**(重抽/抽错不过锚就不采纳)。
- 落点:`src/agents/extract_heal_agent.py`(新)+ `table_scanner` 暴露"单页可调参重抽"入口。**最深,最后做。**

## 六、两个支柱

### 6.1 固化 = 经验库(越跑越强)

每次成功自愈产出**可复用产物**,下次同模式免 LLM:

| 自愈层 | 产物 | 已有? |
|---|---|---|
| L2 改规则 | 规则版本(`parser_rules/versions/`) | ✅ `save_version` |
| L3 抽表/代码 | 认证解析器(`parser_catalog` + `code_generator` + `certify`) | ✅ 构建期闭环已有；`run_field` 经 `route_field` 消费 |
| 选表 | 该版式的选表标记/负分 | 待做 |

`consolidate()` 统一负责把成功的 healer 产物落库,并在下次通过 `route_field`(routed)直接命中。这是"修一次永久会"的关键。

### 6.2 双闸 = 两个**独立真值**锚(过锚 + 复核)

正确率的底气来自**两个不同层面的独立真值**,healer 产物两个都过才入库:

| | **金额锚**(数字真值) | **源文锚 / 复核**(文本真值) |
|---|---|---|
| 代码 | `field_plausibility`(过锚闸) | `verify_field`(复核闸) |
| 真值 | 外部数字:利润表营业收入 | 原始表格文本:源文逐格 |
| 判什么 | 分项**和** ≈ 口径(结构/合计对不对) | **逐项**核 + **表身份/完整性**(选对表没、够全没) |
| 性质 | 确定性、快、可复现 | LLM、需 prompt 调、天生较脆 |

**互补盲区**:只有金额锚 → 和对但单项抠错(取到隔壁列)照样入库;只有复核 → LLM 会被坑表骗、会漏判。两个独立真值都过才入库,一个被骗/失明另一个兜住。徐工"10%以上"坑表就是**双重拦截**(金额锚:和不够 + 复核:标题10%以上)。所以能**大胆堆 healer**,正是这两锚兜底(`_is_ten_pct_trap` + 复核补坑规则)。

#### ⚠ 已知缺陷:金额锚的"口径"错了——分项不必等于营业收入

**会计准则不要求构成表列全金额到 = 营业收入。**
- **营业收入 = 主营业务收入 + 其他业务收入**。格式准则第2号里维度构成表披露的是"**主营业务**分行业/分产品/分地区"(标题即主营),`Σ分项 = 主营 ≤ 营收`,差的是其他业务收入(常不按维度拆);
- "占营业收入 **10%以上**的行业/产品/地区"表**天生只列大项、不完整**;
- 新收入准则 14 号的收入分解,类别由管理层自选,不强制每个口径 = 营业收入。

**后果**:现在金额锚 hard-require `Σ ≈ 营业收入(±3%, all dims)` 会**假失败**——其他业务收入 >3% 的公司(地产/贸易/材料多的制造业),正确的主营构成表被误杀。失败清单里"单维不齐 / 口径差 / 部分严重缺"有一批就是**准则允许的合法主营披露**,不是解析错。

**修正:金额锚改双口径**。一个维度 `Σ ≈ 营业收入` **或** `Σ ≈ 主营业务收入`(各 ±3%)即算过锚。
- 主营锚来源:附注"营业收入构成:主营/其他/合计"表直接抽;或 `营收 − 其他业务收入`;取不到时退单边容差(落在 `[营收×(1−其他业务上限), 营收]` 也算)。落点 `src/eval/anchors.py` + `field_plausibility::_attach_confidence`。
- 与选表 agent 早有的 `caliber_gap`("主营略小于营收=口径差仍算对表")**对齐**——现在确定性金额锚比 LLM 选表还严,不一致。

**为什么放宽是安全的**:口径放宽后怕坑表(也是"主营、不完整")混入——但**表身份交给源文锚(复核)守**(标题"10%以上"+毛利率列已挡)。分工:**金额锚判口径合理(营收/主营),源文锚判表对不对、全不全**。单松金额锚会放错,单靠复核会漏判,两个一起才对。

## 七、终态与度量

- **终态**:`committed`(自主入库)/ `no_such_table`(真无表)/ `out_of_scope`(金融域外)/ `needs_human`(带明确病因)。前三种都算"自愈成功处理",只有 `needs_human` 才是真正待人工。
- **度量**:`/console/failures` 页(`failure_analysis`)已能实时按类拆解;每推进一个 healer,对应类别数应下降、`committed` 应上升。**全自愈率 = (committed + no_such_table + out_of_scope) / 全部**。

## 八、落地路线图

0. **金额锚双口径(地基,先做)** —— 见 6.2:金额锚从"Σ≈营业收入"改成"≈营收 **或** ≈主营"。这**改变了什么算失败**——会先回收一批"合法主营披露被误杀"的假失败(单维不齐/口径差),让后面的诊断/路由建立在正确的失败集上。落点 `anchors.py` + `field_plausibility`。
1. **Tier 1 诊断路由器(骨架)** —— 把 `_classify_failure` 升级为路由器,`_nongreen_llm` 改为"按类派 healer",`judge_diagnose.next_action` 真正映射并执行 healer。做完现有 healer(选表/L2/跨页/避坑)的触达率立刻上来("全空 8"就有救)。
2. **取锚自愈 + 金融股域外** —— 砍"无锚"整类。
3. **选表多轮 + L2 迭代 + 认列扩单位/名称** —— 加深现有 healer。
4. **抽表自愈 L3 + 固化认证解析器** —— 攻真漏行 + 建经验库。
5. **Tier 2 超级管家** —— 先接"批量复盘驱动路线图"(离线、最安全、最快见价值),再接"链尽/歧义时的兜底编排"。放在有了几个 healer、失败尾巴变清晰之后做,管家才有的可管、有的可复盘。

每步落地后:重跑 100 家、看 `/console/failures` 的类别下降与 `committed` 上升,再进下一步。全自愈率 = (committed + no_such_table + out_of_scope) / 全部。
