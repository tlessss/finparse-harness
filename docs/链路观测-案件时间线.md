# 链路观测 · 案件时间线

> **结论**：后端 `run_field(use_llm=True)` 已能跑完整自愈链，但前端**无法一眼看清「这份报告实际发生了什么」**。观测（observability）与正确率、自主率同等重要——看不清就没法调 healer、没法复盘批跑、没法接超级管家。
>
> 本文档记录问题诊断、目标交互、数据模型与分期落地。与 [`Agent深度化-执行计划.md`](Agent深度化-执行计划.md) Phase 4（事件流）对齐并扩展为**生产 Pipeline 全链路**观测。

---

## 一、为什么要做

1. **调参 / 自愈依赖可见性**：L1 选表、L2 改规则、L3 重抽、诊断——哪步试了、为何不过闸，必须可追溯。
2. **批跑复盘**：100 家里 `verify_hold` 长尾，需要按「实际路径」聚类，而不是只看最终 `outcome`。
3. **人机协作**：交人工时要带**病因 + 已试 healer 轨迹**，否则审核台重复劳动。
4. **超级管家前置**：Tier 2 管家读「这份已经试过什么」——没有事件流，管家无输入。
5. **面试 / 对外演示**：「选择即验证 + 多层自愈」需要一条可点击的时间线，而不是静态架构图。

**原则**：观测不削弱双闸——事件只**记录**过锚/复核结论，不改变入库逻辑。

---

## 二、现状：三套 UI、两套数据、对不上

### 2.1 前端三页各讲一半

| 页面 | 路径 | 展示 | 缺口 |
|------|------|------|------|
| 解析流程图 | `/console/flow` | 静态决策图（所有可能路径） | 不知**本案**走了哪条 |
| 流水线成功率 | `/console/pipeline` | 确定性 4 步 + 最多 2 段 LLM Pass | L2/L3/诊断/routed 几乎不展示 |
| 失败分析 | `/console/failures` | 结局 + 类别 | 无过程，只链到 pipeline |

另有 **测试 Tab**（JudgeTest / HealTest / VerifyTest）与 Pipeline **各跑各的**，无统一 `run_id`。

### 2.2 后端：跑了什么 vs 存了什么

**实际链路**（`src/pipeline.py` · `run_field(use_llm=True)`）：

```text
route_field? → verify → [选表自愈] → [L2 rule_heal] → [L3 extract_heal] → judge_diagnose → committed | human
     ↑ 冷启动回落
非绿灯 → _diagnose_category → 按类派 healer → …
```

**`field_chain()` 只算确定性前半段**（给前端 `chain.stages`）：

```text
抽表 → 选表 → 解析 → 锚判(逐维) → outcome
```

**`pipeline_runs.verify`** 是扁平 JSON blob，`save_verify_run` 只摘部分字段；`rule_heal`、`extract_heal`、`healed_rule`、`rule_version_saved`、`via` 等**未完整持久化**，前端画不全时间线。

### 2.3 用户要看清一家公司的成本

当前需：失败分析 → 点公司跳 pipeline → 看 4 步 → 可能再点「跑复核 agent」→ 另开流程图对照理论路径。**多页拼凑**，不符合「一眼看清」。

## 三、目标：案件时间线（Case Timeline）

### 3.1 核心交互

统一入口（建议）：

```text
/console/case?code=300034&year=2025&field=revenue_breakdown
```

点任意公司（流水线列表 / 失败分析 / 分诊队列）→ **同页一条竖向时间线**：

```text
09:41:02  routed          rev_300034 认证解析器命中
09:41:05  verify          verdict=hold · wrong_table
09:41:18  heal_select     重选 p42「营业收入构成」
09:41:22  reparse         过锚 high
09:41:35  verify          verdict=hold · 缺行
09:41:40  rule_heal       delta → rev_heal_300034_2025.yaml
09:41:52  verify          verdict=pass
09:41:52  committed       入库测试库
```

- 点击任一步 → 展开：表预览、解析 JSON、agent 对话、模型、耗时。
- 右侧可选：**流程图高亮本案路径**（`PipelineFlow` 增加 `activePath` 模式：理论图 | 本案路径）。

### 3.2 与超级管家的关系

| 能力 | 观测层 | 管家层 |
|------|--------|--------|
| 记录每步 agent 输入/输出 | ✅ 事件流 | 读事件流 |
| 决定下一步 healer | — | ✅ |
| 改 prompt YAML | — | ❌（只建议，人批） |
| 过锚 + 复核双闸 | 记录 verdict | 不绕过 |

**观测优先于管家**：先有时间线，再接 `steward_agent` 读轨迹决策。

## 四、数据模型

### 4.1 表 `process_events`（SQLite · `goldset/test_store.db`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 自增 |
| `run_id` | TEXT | 一次 `run_field(use_llm=True)` 或调试台会话 UUID |
| `created_at` | TEXT | ISO 时间 |
| `stock_code` | TEXT | |
| `year` | INT | |
| `field` | TEXT | |
| `agent_id` | TEXT | 如 `verify` / `select_table` / `rule_heal` / `extract_heal` / `judge_diagnose` / `pipeline` |
| `event_type` | TEXT | 见下表 |
| `outcome` | TEXT | 本步局部结局：ok / fail / skip / pass / hold |
| `payload_json` | TEXT | 结构化详情（见 4.3） |

索引：`(run_id, id)`、`(stock_code, year, field, id DESC)`。

`pipeline_runs` 增加可选列 `run_id`，与最新事件关联。

### 4.2 生产 Pipeline 事件类型（`event_type`）

| event_type | 触发点 | 典型 payload |
|------------|--------|----------------|
| `input_ready` | `_ensure_report_input` | tables 数、是否新下载 PDF |
| `domain_scope` | 金融域外提前返回 | reason |
| `routed` | `route_field` 命中 | parser_id、confidence |
| `cold_parse` | 冷启动 / 版本池 / 跨页 | version、version_sweep、stitched_pages |
| `anchor_check` | `field_plausibility` | per_dim、confidence、routed_cat |
| `verify` | `verify_field` | verdict、suspects、summary |
| `heal_select` | `heal_select` / `_heal_and_verify` | chosen_page、outcome、select_stage |
| `rule_heal` | `_rule_heal_and_verify` | delta、rounds、rule_version_saved |
| `extract_heal` | `_extract_heal_and_verify` | profile、settings、tries |
| `diagnose` | `_run_diagnose` | decision、root_cause、next_action |
| `codegen` | `_codegen_and_verify`（若接入） | rounds、certified_parser |
| `committed` | `_auto_commit` | table |
| `human` | enqueue needs_human | note |

调试台事件（Phase 4 原案，保留）：`stage1_prepare | stage1_chat | stage2_prepare | stage2_chat | apply_fix`。

### 4.3 payload 约定

- **必含**：`summary`（一行人话）、`duration_ms`（可选）
- **LLM 步**：`chat: { system, prompt, reply }`（与现 `verify_cached` 一致）
- **表相关**：`page`、`caption`、`source_preview`（截断）
- **不写**：整本 PDF、全表网格（过大）；用 `page` + 已有 `/debug/page` 按需拉图

### 4.4 后端 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/pipeline/timeline?stock_code=&year=&field=` | 最近一条 `run_id` 的全事件；无则回退拼 `chain`+`verify` |
| GET | `/pipeline/runs?stock_code=&year=&field=` | 历史 run 列表（批跑多次） |
| GET | `/pipeline/chain` | 保留；确定性阶段 + `verify_cached` |
| GET | `/pipeline/llm` | 保留；触发新 run 时生成新 `run_id` 并写事件 |

实现：`src/pipeline.py` 内 `emit_event(...)`；`src/eval/test_store.py` 增表与读写。

## 五、前端改造

### 5.1 新组件 `CaseTimeline.tsx`

- 嵌入 `PipelinePanel` 右侧，**替代**「①② 两次 Pass」为主视图（Pass 降为点击 event 后的详情）。
- 节点样式：确定性=蓝、LLM=紫、自愈=琥珀、入库=绿、人工=红。
- 终态横幅：`committed` / `verify_hold` / `no_such_table` + `routed_cat` + `reason`。

### 5.2 `PipelineFlow.tsx` 增强

- 模式：**理论全图** | **本案路径**（根据 timeline 的 `event_type` 高亮 `LINKS` 子集）。
- 从 `/console/case` 带 query 打开时默认「本案路径」。

### 5.3 导航收敛

- `FailuresPanel` / `TriageQueue` / `PipelinePanel` 列表项 → `/console/case?code=…`
- `ConsoleShell` NAV 可增加「案件详情」或保持 pipeline 页内嵌 Timeline。

## 六、分期落地

| 阶段 | 内容 | 预估 | 验收 |
|------|------|------|------|
| **P0** | 扩 `save_verify_run` 字段（rule_heal、extract_heal、healed_*、via…）；`PipelinePanel` 展示 L2/L3/诊断卡片 | 0.5d | 跑批后点 300034 能看到 rule_heal 与 yaml 路径 |
| **P1** | `process_events` 表 + `emit_event` + `GET /pipeline/timeline` + `CaseTimeline` | 1–2d | 一次 `run_full_pass` 单份可见 **≥6 条** event |
| **P2** | `PipelineFlow` 本案路径高亮；`/console/case` 统一入口 | 1d | 理论图与实际路径对照 |
| **P3** | 调试台共用 `run_id`；`process_events` 与 Steward 读取 | 1d | JudgeTest 与 Pipeline 同一时间线 |

**推荐顺序**：P0 → P1 → P2；P3 与 Agent 深度化 Phase 4–5 合并。

## 十、一句话

> **链路观测 = 把 `run_field` 的每一步写成可查询的事件时间线，让前端一眼看清「发生了什么、卡在哪、试过什么」——这是调自愈、接管家、做批跑复盘的前提。**
