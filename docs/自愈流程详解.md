# FinParseAI 自愈流程详解：从诊断到固化

> 一份 A 股年报解析器，怎么在出错时"自己把自己修好"——而且不被一个不靠谱的大模型带偏，也不为同一个错反复花钱。

## 一、为什么需要自愈

一份年报 PDF，几百页，表格千奇百怪：有的"营业收入构成"带占比列，有的是毛利率表；有的写"分销售模式"，有的写"销售模式"；有的单位是元，有的是千元，单位标记还藏在表格上方的正文里。**没有一套通用解析器能吃下所有版式。**

传统做法有两条死路：

- **人肉穷举规则**：每见一种新版式就加一段代码，永远修不完。
- **运行期上大模型**：每份报告都喂 LLM，慢、贵、还不稳——同一份报告判两次能给出不同结果。

FinParseAI 走第三条：**构建期自愈，运行期零 LLM**。大模型只在"构建期"出现——当某份报告解析错了，它帮忙诊断、提修复；修好后把这条规则**固化成全局规则**。等到真正批量跑的时候，运行期是纯确定性的 Python，一个 LLM 都不调。

这篇文章讲中间那个"修"的过程——自愈。它的精髓只有一句：

> **每个错只学一次，修复全局生效，从此再不犯、再不花钱。**

## 二、核心理念：别修没坏的

自愈最大的风险不是"修不好"，而是**"修一个没坏的东西，反而修坏了"**。

一个真实例子。赛力斯（601127）的营收按四个维度拆分（行业/产品/地区/销售模式），每个维度的分项和都是 **1596 亿**，内部完全一致。但数据库里的"营业收入"锚是 **1651 亿**，差了 3.3%。

机器一看"对不上锚"，很容易判定"坏了，去修"。但其实——**两个数都对**。1596 亿是"营业收入"（主营），1651 亿是"营业总收入"（含其他业务、利息等）。这是会计口径的正常差异，不是 bug。如果自愈在这里硬把毛利率(28.21)当成占比去"修复"，只会越修越错。

所以自愈的第一性原理是：

> **判官必须是确定性的（锚 + 维度一致），不是大模型；先准确判断"到底坏没坏"，再谈修。**

## 三、为什么不能让 LLM 单干

我们做过一个实测，结果很说明问题。

把赛力斯（一个**没坏**、只是口径差的案子）连同解析器代码、配置一起喂给 DeepSeek，问它"bug 在哪"。它回答得**非常自信、非常详细**：

> "根因：`section_labels` 缺少 '销售模式' 标签 → 直销/经销被归入 regions → by_channel 为空。"

听起来天衣无缝。但**全错**：当前代码**有** "销售模式"（早就加过了，喂给它的就是这份最新代码）；by_channel **不空**；这份数据根本**没坏**。它凭空诊断出一个早已修好的 bug，连症状都编错了。

为什么？**确认偏误**。你让大模型"找 bug"，它就一定给你找一个出来，哪怕没有，也编得有鼻子有眼。

这次"事故"反而钉死了整套架构的合理性：**LLM 不能单独决定任何事**。它必须被夹在两道确定性的闸门中间——进门要筛子放行，出门要锚验收。

## 四、整条链：五步干净闭环

```
报告解析出错
    │
 ① 真失败筛子 (零LLM)
    │   任一维度过锚        → 无需自愈
    │   各维度一致但偏锚    → 口径差，非bug，不碰
    │   维度互相矛盾        → 真bug ↓
    │   无锚                → 转人审
    │ (只有"真坏"才往下，且每个错一辈子只走一次)
    │
 ② LLM 诊断 + 出修复 (LLM在这，但被夹住)
    │   给 AI 调试包：病历 + 原表 + 错值 + 锚 + 当前配置 + 相关代码
    │   → AI 指根因 + 提最小修复（优先"改哪条配置/规则"）
    │
 ③ 应用为【全局规则】(分层)
    │   Tier1 规则(默认): 改共享配置 → 所有报告立即生效
    │   Tier2 代码(兜底): 规则表达不了才写Python, 进沙箱
    │
 ④ 回链重测 + 回归验证 (零LLM)
    │   parse → 锚 →(判定)，过锚才算修好
    │   再跑一遍历史回归集，确认没修坏别的
    │
 ⑤ 固化 + 入库 (人兜底)
        过 → 规则永久生效 / 指纹认证 + 人审 → 入库
        不过 → 升级(规则→代码) / 转人工
```

### ① 真失败筛子

判官是**锚（分项和 ≈ 营业收入）+ 维度一致（各维度的和应该相等，因为它们是同一总额的不同切法）**。任一维度过锚 → 无需自愈；各维度一致但偏锚 → 口径差，不碰；维度互相矛盾 → 真 bug；无锚 → 转人审。

这一步是 LLM 的**准入门**，挡住假警报。而且——**同一个错只会触发它一次**，修好固化后这个错就消失了，下次根本走不到这一步。

### ② LLM 诊断 + 出修复

喂给 LLM 的是一个**聚焦的调试包**：病历（确定性诊断）+ 失败案例（原表 / 错值 / 锚）+ 当前配置 + 相关解析器代码。让它像资深开发者那样**看代码 + 看失败输入 + 看错误输出**，定位根因、提最小修复（**优先回答"能不能只加/改一条配置"**）。

关键约束：给聚焦上下文（不是整个仓库）；优先改配置/规则（不到万不得已不改代码）；这是**可编辑的对话**，人能改 prompt、反复试、全程留痕。

### ③ 应用为全局规则（这一步是整套省钱的关键）

```
Tier 1 规则（默认，~80%）
   add_section_marker / set_unit / add_header_alias …
   改的是【共享配置】，对所有报告立即生效

Tier 2 代码（兜底，~20%）
   规则表达不了的新逻辑才写 Python，进沙箱
```

**为什么修复必须是全局的——这是自愈不烧钱的根本。** 把"销售模式"加进共享的 `section_labels` 后，之后**任何**写"销售模式"的报告全都自动正确，这个错**再也不会犯、再也不会叫 LLM**。修复落在共享规则上，一次投入、永久收益。

### ④ 回链重测 + 回归验证

改完，**回到同一条链**重跑：parse → 锚 →（判定）。**过锚才算修好。** 这道闸兜住 AI 的一切幻觉——前面那个"赛力斯幻觉 bug"，就算你信了它去改，回链重测发现数据没变好，自动否决。

再加一步**回归验证**：因为这次改的是全局规则，要在一批历史正确的报告上重跑，确认"没把别的修坏"。全局规则的代价是要防回归，回归集就是保险。

### ⑤ 固化 + 入库

过 → 规则永久生效（需要时按指纹认证专用解析器）+ 人审 → 入库。不过 → 升级（规则 → 代码）或转人工。**人是写进生产库前的最后一闸。**

## 五、核心洞察：LLM 调用量按"错误种类"封顶

这是整套设计最反直觉、也最值钱的一点。

很多人以为"几千份报告就要几千次 LLM"。**错**。因为修复是全局规则：

```
"销售模式漏标记" 这个错：
  第 1 次遇到（赛力斯） → LLM 诊断 → 加全局规则 → 验证 → 固化
  第 2 ~ 5000 次遇到    → 规则已生效，直接对，零 LLM
```

所以 **LLM 调用量 = 不同错误的种类数，与报告数量无关**。

- 跑得越多，剩下的**新错**越少，LLM 越来越闲；
- 几千份报告的失败，收敛到的"错误种类"也就几十到上百个量级（按变体算，`销售模式`/`按销售模式`/`销售渠道` 各算一种）；
- 最终逼近"运行期零 LLM"——不是因为不用大模型，而是因为**该学的都学完了**。

这也顺带解释了：为什么不必预先搭一套复杂的"确定性自动修复引擎"去省 LLM——既然每个错总共就调一次 LLM，专门写代码去省这一次，性价比是负的。**让系统按数据自然变聪明，比预先过度设计更划算。**

## 六、三条铁律

1. **只喂真坏的给 LLM。** passing / 口径 的案子去问 AI"哪错了"，确认偏误会让它必然幻觉。筛子（①）是 LLM 的准入门。
2. **LLM 提名，确定性判官裁决。** 锚守门（①）、锚验收（④），LLM 夹在中间，不掌生杀。
3. **人是入库最后一闸。** 再自动，写进生产库前必有人点头。

## 七、两个真实案例，正反对照

| | 赛力斯 601127（假警报） | 销售模式漏标记（真 bug） |
|---|---|---|
| 现象 | 各维度都 1596 亿，与锚 1651 亿差 3.3% | regions = 3192 亿 = 正常的 2 倍 |
| 筛子判定 | 各维度一致 → **口径差，不自愈** | 维度矛盾 → **真 bug，进自愈** |
| 真相 | 1596(营收) 和 1651(营业总收入)都对 | 销售模式(直销/经销)漏标记，串进了 regions |
| 该怎么做 | **不碰**（碰了只会修坏） | 加全局规则 `add_section_marker("销售模式")` → 重测过锚 → 固化 |
| 之后 | —— | 全部 5000 份里写"销售模式"的，**一次全好，再不犯** |

两个案子一对照，整套逻辑就活了：**判官分得清真假，自愈只对真的动手，修一次全局永久生效。**

## 八、现状与缺口

```
① 真失败筛子        ✅ 已建（heal_debug / ⑥ 自愈测试页）
② LLM 诊断+修复     ✅ 已建（heal_prepare/heal_chat，给代码+配置的可编辑对话）
③ 应用为全局规则    ❌ 缺口——还没建规则工具（add_section_marker 等）
④ 回链重测+回归     🔸 锚现成，回归集验证还没接
⑤ 固化 + 入库       ✅ 已建（CommitReview 入库审核 + 认证链）
```

**最大的缺口在 ③——工具。** 现在 AI 能说"加一条 '销售模式' 标记"，但还得人手动去改配置。下一步把 **② → ③ → ④ 接通**：AI 出结构化修复（改哪条配置）→ 一键应用成全局规则 → 自动回链重测 + 回归 → 过了才固化。先建第一个规则工具 `add_section_marker`，拿一个真坏的 case，跑通"AI 提议 → 应用 → 重验 → 全局生效"的完整闭环。

## 九、结语

自愈不是"让 AI 修代码"那么简单。它是一套**用确定性守门、把不可靠的大模型夹在中间、靠人兜底、让修复全局生效的工程**：

> 锚判断真假，LLM 提供线索，工具改成全局规则，锚再验收，人最后把关。

大模型在这里不是英雄，是个**有用但靠不住的实习生**——你给它聚焦的材料，听它的建议，但每一步都验证，绝不让它单独签字。这样即便用着一个会幻觉的弱模型，系统整体依然可靠。

而每修好一类错，就固化成全局规则。**LLM 的调用量被错误的种类数封顶，跟报告数量无关**——跑得越多，会得越多，新错越少，需要 LLM 的地方越少，最终逼近"运行期零 LLM"的纯确定性解析。这，就是自愈的意义。
