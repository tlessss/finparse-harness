# FinParseAI 自愈流程详解：从诊断到入库

> 一份 A 股年报解析器，怎么在出错时"自己把自己修好"——而且不被一个不靠谱的大模型带偏。

## 一、为什么需要自愈

一份年报 PDF，几百页，表格千奇百怪：有的"营业收入构成"带占比列，有的是毛利率表；有的写"分销售模式"，有的写"销售模式"；有的单位是元，有的是千元，单位标记还藏在表格上方的正文里。**没有一套通用解析器能吃下所有版式。**

传统做法有两条死路：

- **人肉穷举规则**：每见一种新版式就加一段代码，永远修不完。
- **运行期上大模型**：每份报告都喂 LLM，慢、贵、还不稳——同一份报告判两次能给出不同结果。

FinParseAI 走第三条：**构建期自愈，运行期零 LLM**。大模型只在"构建期"出现——当某份报告解析错了，它帮忙诊断、提修复；修好后把这套规则/解析器**按版式指纹固化**下来。等到真正批量跑的时候，运行期是纯确定性的 Python，一个 LLM 都不调。**学一次，永久会。**

这篇文章讲的就是中间那个"学"的过程——自愈。

## 二、核心理念：别修没坏的

自愈最大的风险不是"修不好"，而是**"修一个没坏的东西，反而修坏了"**。

一个真实例子。赛力斯（601127）的营收按四个维度拆分（行业/产品/地区/销售模式），每个维度的分项和都是 **1596 亿**，内部完全一致。但数据库里的"营业收入"锚是 **1651 亿**，差了 3.3%。

机器一看"对不上锚"，很容易判定"坏了，去修"。但其实——**两个数都对**。1596 亿是"营业收入"（主营），1651 亿是"营业总收入"（含其他业务、利息等）。这是会计口径的正常差异，不是 bug。如果自愈在这里硬把毛利率(28.21)当成占比去"修复"，只会越修越错。

所以自愈的第一性原理是：

> **判官必须是确定性的（锚 + 维度一致），不是大模型；先准确判断"到底坏没坏"，再谈修。**

## 三、为什么不能让 LLM 单干

我们做过一个实测，结果很说明问题。

把赛力斯（一个**没坏**、只是口径差的案子）连同解析器代码、配置一起喂给 DeepSeek，问它"bug 在哪"。它回答得**非常自信、非常详细**：

> "根因：`section_labels` 缺少 '销售模式' 标签 → 直销/经销被归入 regions → by_channel 为空。"

听起来天衣无缝。但**全错**：

- 当前代码**有** "销售模式"（早就加过了，喂给它的就是这份最新代码）；
- by_channel **不空**，明明有直销/经销两项；
- 这份数据根本**没坏**。

它凭空诊断出一个早已修好的 bug，连症状都编错了。

为什么？**确认偏误**。你让大模型"找 bug"，它就一定给你找一个出来，哪怕没有，也编得有鼻子有眼。

这次"事故"反而钉死了整套架构的合理性：**LLM 不能单独决定任何事**。它必须被夹在两道确定性的闸门中间——进门要筛子放行，出门要锚验收。

## 四、整条链：六个阶段

```
                    ┌─────────────── 真失败筛子 (零LLM) ───────────────┐
报告解析出错 ──→ ⑥ 阶段0：到底坏没坏？                                │
                    │   任一维度过锚        → 无需自愈                  │
                    │   各维度一致但偏锚    → 口径差，非bug，不碰        │
                    │   维度互相矛盾        → 真bug ↓                   │
                    │   无锚                → 转 LLM/人审               │
                    └──────────────────────────┬──────────────────────┘
                                                │ (只有"真坏"才往下)
                    阶段1：精确定位 (零LLM)      │
                       哪个维度坏 / 哪一环 / 应该是啥 → 修复说明书
                                                │
                    阶段2：出修复方案 (LLM在这，但被夹住)
                       给 AI 调试包(病历+原表+错值+锚+配置+代码)
                       → AI 指根因 + 提最小修复(优先改配置)
                                                │
                    阶段3：应用修复 (分层)
                       Tier1 规则工具(默认): 改配置, 安全可审计
                       Tier2 代码(兜底): 规则表达不了才写Python, 进沙箱
                                                │
                    ┌──────────── 回链重测 (零LLM) ────────────┐
                    阶段4：parse → 锚 →(判定)                    │
                       过锚才算修好 ← 这道闸兜住AI的一切幻觉      │
                    └────────────────┬─────────────────────────┘
                                     │
                    阶段5：收尾
                       过 → 认证(指纹→规则/解析器) + 人审 → 入库
                       不过 → 升级(规则→代码) / 转人工
```

### 阶段 0｜真失败筛子

判官是**锚（分项和 ≈ 营业收入）+ 维度一致（各维度的和应该相等，因为它们是同一总额的不同切法）**。

- **任一维度过锚** → 数据可信，无需自愈。
- **都不过、但各维度互相一致** → 多半是口径差（营收 vs 营业总收入），非 bug，不碰。
- **维度互相矛盾**（比如某维度 ≈ 另一维度的 2 倍）→ 真 bug，往下走。
- **无锚**（客户/供应商/员工没有 DB 锚）→ 不走确定性，转 LLM/人审。

这一步是 LLM 的**准入门**。挡住假警报，是整套不翻车的前提。

### 阶段 1｜精确定位

把"维度矛盾"细化成一张能动手的**修复说明书**：

- **哪个维度坏？** 比各维度的和，找离群的那个。
- **坏在哪一环？** 回查它的认列轨迹、选表、切桶——是表选错了、列认错了、还是桶切漏了。
- **应该是啥？** 拿正常维度的和、或锚，当**校验靶子**。

这一步纯确定性，不烧 LLM，但它给后面的 AI 聚焦——少看无关的东西。

### 阶段 2｜出修复方案

这是 LLM 唯一上场的地方，但喂给它的是一个**聚焦的调试包**：

```
病历（确定性诊断）+ 失败案例（原表 / 解析出的错值 / 锚）
+ 当前配置（revenue.yaml 相关段）
+ 相关解析器代码（认列 / 切桶函数源码）
```

让它像一个资深开发者那样**看代码 + 看失败输入 + 看错误输出**，定位根因、提最小修复（**优先回答"能不能只加/改一条配置"**）。

关键约束：**给聚焦上下文，不是整个仓库**；**优先让它改配置/规则，不到万不得已不改代码**。而且这是一个**可编辑的对话**——人能改 prompt、反复试、全程留痕，找到最稳的修复写法。

### 阶段 3｜应用修复：两层，规则优先

```
Tier 1 规则工具（默认，~80%）
   add_section_marker / set_unit / add_header_alias …
   改的是配置，安全、可审计、改不坏

Tier 2 代码生成（兜底，~20%）
   规则系统表达不了的新逻辑才写 Python，进沙箱跑
```

我们修过的 bug 几乎全是规则级的（销售模式没切桶、单位检测、合计行混入），所以**规则工具能覆盖大多数**。代码那条留给真正的长尾。两层都有，但**规则优先、代码兜底**——能改配置就不写代码。这样既不为 80% 的小修付代码的风险，也不为 20% 的硬骨头丢代码的能力。

### 阶段 4｜回链重测：最重要的一道闸

改完，**回到同一条链**重跑：parse → 锚 →（判定）。

> **过锚才算修好。**

这道闸是整套能用弱模型的**根本**。无论 AI 的修复建议听起来多漂亮，应用之后数据没变好，就直接否掉。它兜住 AI 的一切幻觉——前面那个"赛力斯幻觉 bug"，就算你信了它去"加销售模式"，回链重测发现数据没变（本来就对），自动否决。

### 阶段 5｜收尾

- **过** → 按版式指纹**认证**（指纹 → 规则集 / 解析器）+ **人审** → 入库。
- **不过** → 升级（规则 → 代码）或转人工。

认证后，下次遇到**同版式**的报告，直接路由命中、零 LLM。这就是"自愈"的终点——不是修好这一份，而是**让系统永久学会这一类**。

## 五、三条铁律

吃过亏才总结出来的：

1. **只喂真坏的给 LLM。** passing / 口径 的案子去问 AI"哪错了"，确认偏误会让它必然幻觉。筛子（阶段 0）是 LLM 的准入门。

2. **LLM 提名，确定性判官裁决。** 锚守门（阶段 0）、锚验收（阶段 4），LLM 夹在中间，不掌生杀。

3. **人是入库最后一闸。** 再自动，写进生产库前必有人点头。LLM 觉得没问题 → 浅绿色待审 → 人通过 → 才入库。

## 六、两个真实案例，正反对照

| | 赛力斯 601127（假警报） | 销售模式漏切桶（真 bug） |
|---|---|---|
| 现象 | 各维度都 1596 亿，与锚 1651 亿差 3.3% | regions = 3192 亿 = 正常的 2 倍 |
| 筛子判定 | 各维度一致 → **口径差，不自愈** | 维度矛盾 → **真 bug，进自愈** |
| 真相 | 1596(营收) 和 1651(营业总收入)都对 | 销售模式(直销/经销)漏标记，串进了 regions |
| 该怎么做 | **不碰**（碰了只会修坏） | add_section_marker("销售模式") → 重测过锚 → 认证 |

两个案子一对照，整套逻辑就活了：**判官分得清真假，自愈只对真的动手。**

## 七、现状与缺口

```
阶段0 真失败筛子      ✅ 已建（heal_debug / ⑥ 自愈测试页）
阶段1 精确定位        🔸 部分（诊断有了，"修复说明书"还没结构化）
阶段2 AI 出修复方案   ✅ 已建（heal_prepare/heal_chat，给代码+配置的可编辑对话）
阶段3 应用（工具）    ❌ 缺口——还没建规则工具（add_section_marker 等）
阶段4 回链重测        ✅ 锚现成（parse / heal_debug）
阶段5 认证 + 入库     ✅ 已建（CommitReview ⑤ 入库审核 + 认证链）
```

**最大的缺口在阶段 3——工具。** 现在 AI 能说"加一条 '销售模式' 标记"，但还得人手动去改配置。下一步是把**阶段 2 → 3 → 4 接通**：AI 出结构化修复（改哪条配置）→ 一键应用 → 自动回链重测 → 过锚才认证。先建第一个规则工具 `add_section_marker` + 阶段 4 自动重测，拿一个真坏的 case，跑通"AI 提议 → 应用 → 重验"的完整闭环。

## 八、结语

自愈不是"让 AI 修代码"那么简单。它是一套**用确定性守门、把不可靠的大模型夹在中间、靠人兜底的工程**：

> 锚判断真假，LLM 提供线索，工具执行修复，锚再验收，人最后把关。

大模型在这里不是英雄，是个**有用但靠不住的实习生**——你给它聚焦的材料，听它的建议，但每一步都验证，绝不让它单独签字。这样即便用着一个会幻觉的弱模型，系统整体依然可靠。

而每修好一类版式，就用指纹固化下来。跑得越多，会得越多，需要 LLM 的地方越少——最终逼近"运行期零 LLM"的纯确定性解析。这，就是自愈的意义。
