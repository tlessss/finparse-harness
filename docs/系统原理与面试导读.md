## 0. 一分钟讲清（电梯陈述）

> "我做的是一个**会自我改进的 A 股财报解析系统**。难点在于：1200+ 家公司的年报版式各不相同，一套规则解析器净通过率只有 8.9%。
>
> 我的核心思路是**把'写解析器'这件事自动化、但把'跑解析器'保持确定性**：
>
> - **构建期**：让 LLM 针对'一份版式'写一个专用解析器，用**确定性闸门**（对一份人工确认的标准答案做到完全一致 exact）验收，人在回路认证入库；
> - **运行期**：**零 LLM**，跑冻结的确定性 Python。来一份新报告，不去'预测'哪个解析器适用，而是把候选都跑一遍、用**硬性勾稽规则**（如各分项占比之和≈100%）判断谁解对了——我叫它**'选择即验证'**。
>
> 整套框架对营收/成本/研发/员工/客户/供应商**6 个字段通用**，靠一个'字段插件'抽象支撑，每个字段只是一组不同的'勾稽判据'。"

记住三个数字/名词面试官会追问：**8.9%**（起点）、**选择即验证**、**exact 否则转人工**。

## 1. 四个设计哲学（面试官最爱问"你的设计取舍"）

### ① 选择即验证（selection-is-validation）

**不预测，而是验证。** 不试图用分类器猜"这份报告该用哪个解析器"，而是把候选解析器**都跑一遍**，用**客观硬规则**判断谁的输出是对的（占比和≈100、明细和≈合计…）。

- **为什么**：版式千变万化，预测必然出错且无法自证；而"解对了没有"有**数学上的硬约束**可判，零成本、可解释、不会张冠李戴。
- 代码：`revenue_router.route_field` + `field_plausibility`。

### ② 构建期改代码 / 运行期零 LLM

LLM 只在**离线构建期**写解析器代码；**线上运行跑冻结的确定性 Python**。

- **为什么**：运行期要的是**可控、可复现、便宜、快**。LLM 在线解析每份都可能给不同结果、贵、慢、不可审计。把不确定性关进构建期，用闸门挡住，线上就只有确定性代码。

### ③ 终点恒 = exact，否则转人工

一次修复的终点**必须是对标准答案完全一致**；做不到就**转人工**，绝不收下"部分正确"的半成品。

- **为什么**：正确率优先。半成品入库会污染下游、且无法察觉。宁可转人工也不留错。
- 代码：`code_generator.generate_parser` 的"exact-or-escalate"循环。

### ④ 规范驱动 + 字段插件（机器层字段无关）

解析的"认表/认列/判据"口径来自**会计披露准则第2号**（`docs/规范要点-解析映射.md`）；而路由/缓存/打分/自愈/溯源这些"机器"对字段无关，靠一个 `FieldSpec` 插件描述每个字段的差异。

- **为什么**：让系统从"营收专用"变"6 字段通用"，加一个字段=填一个插件，不动机器层。

## 2. 端到端数据流（追一份报告，带代码位置）

入口：`FinParseAI.run()`（`src/engine_orchestrator.py`）

```
run(pdf, code, year)
 │
 ├─ 1) scan_pdf 一次性抽全表 all_tables（含 page/网格/cell_bbox/section）
 │       —— 6 个字段共享这份表，不重复扫
 │
 ├─ 2) 对每个字段 field：
 │     rev_result = self._route_field(REVENUE, code, year, all_tables)   ← 路由优先
 │     if rev_result is None:                                            ← 没命中
 │         rev_result = 通用解析器.parse(...)                            ← 冷启动兜底
 │     （研发/员工/成本/客户/供应商 同构）
 │
 ├─ 3) 组装 output：每个字段的值 + 溯源 + parse_flags
 └─ 4) （可选）写库
```

`_route_field(spec, code, year, all_tables)`（engine 内，机器层与字段无关）：

```
table_cache.put(code,year,all_tables)        # 把已抽好的表喂进缓存，route 不重扫
rt = route_field(spec, code, year)           # ← 选择即验证，见下
if rt.status == "routed":
    field_value = 解包(rt.result)             # D类富结构{top_clients:[...],total}→取列表
    溯源 = attach_provenance(field_value, all_tables, spec)   # 事后反查 bbox
    return {spec.field: field_value, 溯源, _routed:True}
return None                                   # 任何异常都安全回退冷启动
```

`route_field(spec, code, year)`（`src/parsers/revenue_router.py`）——**选择即验证 + 指纹缓存路由**：

```
① 缓存命中：fingerprint_of(code,year) → route_get(field,fp) 拿到上次选中的解析器
            跑它 → field_plausibility 干净？→ 直接 routed（连候选都不跑）
            不干净 → route_invalidate（版式漂移，失效重选）
② 缩候选选优：candidates_for(field, fp)  # 按字段过滤 + 指纹缩候选；没匹配→该字段全跑兜底(召回)
            逐个跑 → field_plausibility 打信号 → 取"干净且行数最多"的
            选中 → route_set(缓存) + tag_fingerprint(自学这个指纹)→ routed
            都不干净 → needs_repair（交给冷启动/自愈）
```

`field_plausibility(spec, value)` —— **运行期判"解对没"的硬规则代理**（无 golden 时用）：按 `spec.cls` 分派 A/B/C 判据（见第 4 节）。

跑某个版本解析器：`version_parse_fn(path)`（`src/eval/sandbox_exec.py`）→ `importlib` 加载 `parse(tables)` → 喂缓存表 → 出结果。

## 3. 文件地图（按层，每个一句话）

**编排层**

- `engine_orchestrator.py` —— 入口 `run()`；抽表一次、6 字段路由优先+冷启动兜底、组装输出。

**路由层（选择即验证）**

- `parsers/revenue_router.py` —— `route_field`（缓存→缩候选→验证选优）、`field_plausibility`（A/B/C 判据信号）。
- `eval/route_index.py` —— `fingerprint_of`（版式指纹，缓存）、`route_get/set/invalidate`（指纹→解析器 的路由缓存，按字段隔离）。
- `eval/parser_catalog.py` —— 认证清单（`certify`/`load_certified`）、`candidates_for`（按字段+指纹缩候选）、`pick_mother`（构建期用 golden 选母本）、`tag_fingerprint`（路由成功后自学指纹）。

**字段插件层（机器层字段无关的关键）**

- `eval/field_spec.py` —— `FieldSpec` 数据类 + 6 个字段实例 + `as_dims`（把"多维字典/扁平列表"统一成 `{维度:[行]}`）。每个 spec 带：amount_key/dims/cls(判据类)/total_key/detail_key + **准则口径**(spec_note/categories/table_markers)。

**验证地基（"自修复的安全上限"）**

- `eval/revenue_score.py` —— `score_field`（对 golden 打分，A/B/C 分派；返回 exact/score/mismatches）。**构建期闸门**。
- `eval/run_eval.py` —— `eval_version`（跑遍 golden 打分）、`accept_candidate`（新版本必须**整体升且零退步**才收）。
- `validators/hard_rules.py` —— `check_hard_rules`（红线勾稽：red 级违规一律判错、阻断入库）。**生产期闸门**。
- `eval/sandbox_exec.py` —— 把解析器 .py 变成可跑的 `parse_fn`（当前 importlib 本进程；硬化 TODO=子进程+超时）。
- `eval/table_cache.py` —— 抽表缓存（`get_tables`/`put`），避免重复扫 PDF。

**自愈层（构建期改代码）**

- `agents/heal_pipeline.py` —— `heal_field`：路由→修复→认证 或 转人工，串成一条，产出统一 heal 记录。
- `agents/code_generator.py` —— `repair`（三岔：复用/fork/新建）、`generate_parser`（LLM 写代码的 exact-or-escalate 循环，**规范驱动 prompt**）、`_feedback`（诊断性负反馈）。
- `agents/llm_client.py` —— OpenAI 兼容客户端（接 DeepSeek）。

**输出增强**

- `eval/provenance.py` —— `attach_provenance`：拿解析出的值回源表里按值匹配，反查 `(page,bbox)`，给"只吐值"的解析器**白补溯源**（人审红框对照原文）。

**解析器本体**

- `parsers/versions/*.py` —— 一份版式一个的**专用解析器**，全实现同一契约 `parse(tables)->字段结构`。如 `rev_000425_v1`(营收)、`cst_300005_v1`(成本)、`rnd_300005_v1`(研发)、`emp_300005_v1`(员工)、`cli_300009_v1`/`sup_300009_v1`(客户/供应商)。
- `parsers/<field>/default.py` —— 通用解析器（冷启动兜底，全文盲扫，准确率低但有总比没有强）。
- `parsers/infra/table_scanner.py` —— 抽表底座（`scan_pdf`/`parse_money`/`parse_ratio`/`is_total_row`/`filter_by_signature`）。

**持久状态**（`goldset/`，gitignore，运行时产物）

- `certified_parsers.json`（认证清单）、`revenue_golden.json`（值级标准答案，多字段）、`route_cache.json`（指纹→解析器）、`fingerprint_cache.json`、`tables_cache/`（抽表缓存）。

## 4. 四个判据类（A/B/C/D）—— 字段插件的核心

不同字段"对不对"的判据不同。`FieldSpec.cls` 选判据，`score_field`/`field_plausibility` 按 cls 分派。


| 类           | 字段     | 数据形状                                      | 判据（勾稽）                               | 代码                                         |
| ----------- | ------ | ----------------------------------------- | ------------------------------------ | ------------------------------------------ |
| **A** 占比构成  | 营收、成本  | 多维字典 / 扁平列表                               | 各维度**占比之和≈100%**                     | `score_field` 主体 / `field_plausibility` 主体 |
| **B** 明细对合计 | 研发     | `{total_this, rnd_detail:[…]}`            | 明细金额**之和≈合计**                        | `_score_b` / `_plaus_b`                    |
| **C** 分项对总数 | 员工     | `{total, composition:[…], education:[…]}` | 各维度人数**之和=总数**                       | `_score_c` / `_plaus_c`                    |
| **D** 弱勾稽   | 客户、供应商 | `{top_*:[…], total_ratio_pct}`            | 前五明细占比**之和≈前五合计**（**复用 B**）；明细缺失→转人工 | 复用 `_score_b`/`_plaus_b`                   |


**关键洞察（面试可讲）**：D 类（客户/供应商）数学上等价于 B 类（部分之和≈声明的合计），所以**没新写一类**，直接复用。而准则规定明细名单"鼓励非强制"——很多公司只披露汇总比例（合规），此时无明细可勾稽→**自动判不了→转人工**，这正是 D 类弱判据的真实写照。

`as_dims` 的作用：营收是 `{industries:[…],segments:[…]}`（多维字典），成本是 `[…]`（扁平列表）——两种容器，用 `as_dims` 统一成 `{维度:[行]}`，让 A 类打分器一套代码同时处理。

## 5. 自愈闭环（构建期改代码，code_generator + heal_pipeline）

当路由 needs_repair 且有 golden（认证期）：

```
heal_field(spec, code, year, golden)            # heal_pipeline.py
 └─ repair(...)                                  # code_generator.py，三岔决策(fork优先)
      ├─ pick_mother：选择即验证，在已认证解析器里对 golden 挑最像的母本
      ├─ 母本 exact(≥0.999)   → 复用(reuse)，直接用，不调 LLM
      ├─ 母本部分(≥0.3)       → fork：把母本源码塞进 prompt，让 LLM 在它基础上改
      └─ 都很差               → 新建：从零写
           └─ generate_parser：exact-or-escalate 循环（最多 N 轮）
                每轮: LLM 出代码 → sandbox 跑 → score_field 打分
                      exact → 收下；否则把 _feedback(诊断+还差哪)喂回去继续
                      N 轮仍不 exact → 转人工(不留半成品)
 └─ exact → certify(key,path,field,指纹)         # 入认证清单 → 下次同版式自动路由
```

**fork 优先的价值**（面试可讲）：弱模型（DeepSeek）从零写常卡住；给它一个"相似版式的已认证母本"让它改，能 1 轮到 exact。**正确性与模型强弱解耦**——闸门挡住错代码，弱模型只是多迭代几轮，不会产出错结果。

**规范驱动 prompt**：`_build_contract(spec)` 把该字段的准则口径（标准类目、目标表特征、章节锚点、避坑）注入 prompt，让 LLM 知道"该找什么表/什么列"，而不是对着表瞎猜。

## 6. 设计决策与权衡（面试金矿，Q&A）

**Q：为什么不直接用 LLM 在线解析每份报告？**
A：贵、慢、不可复现、不可审计，5000+ 家/年规模扛不住。我把 LLM 限制在构建期写代码，运行期跑冻结的确定性 Python——把不确定性关进可验收的离线环节。

**Q：解析器多了，怎么知道一份新报告用哪个？**
A：不预测，选择即验证——候选都跑、用硬勾稽判谁对。为了快，加了**指纹缩候选 + 路由缓存**：同版式第二份直接命中缓存的解析器，连候选都不跑；版式漂移（硬规则不过）则缓存失效、重新选优。

**Q：指纹不准会怎样？**
A：指纹只用来**缩小候选、加速**，不是判对错的依据——判对错永远靠硬规则。所以指纹召回不足时**回退到该字段全部候选都跑**，不会漏掉对的；指纹精度不是承重点。（我实测过 layout_fingerprint 作为路由键过细，所以特意设计成"指纹缩候选 + 硬规则裁决 + 召回兜底"。）

**Q：怎么保证不"张冠李戴"（把 A 公司的解析器用到 B 公司）？**
A：选择即验证天然防错——B 公司的报告就算跑了 A 的解析器，输出过不了硬规则（占比和不≈100）就不会被采纳。实测 000425 的营收解析器对 300005 跑出来不干净→正确 needs_repair，没误用。

**Q：标准答案（golden）怎么来？可信吗？**
A：值级 golden，来自源表、靠**勾稽自证**（明细和≈合计/分项和=总数）。严格说应由人按准则终审——我把这步放在人在回路的认证关。我诚实标注了这是当前的可信度边界。

**Q：让 LLM 改代码，安全吗？**
A：当前 sandbox 用 importlib 本进程跑，适合可信/自己审过的版本；跑任意 LLM 代码要加子进程+超时+资源限制，我列为**批量跑前的安全闸**，和前端控制台一起做，没阻塞闭环打通。

**Q：这套和"一个大规则引擎"比，好在哪？**
A：规则引擎是"一套规则适配所有版式"，必然在长尾版式上崩（净通过率 8.9%）。我是"一份版式一个专用解析器 + 自动生成 + 客观闸门"，把全局难题拆成**可独立验收的局部问题**，爆炸半径只在同版式集合内。

## 7. 怎么 demo / 怎么讲故事

**一句话故事线**：8.9% 净通过 → 发现"一套规则治不了千种版式" → 转向"一版式一解析器 + LLM 自动写 + 客观闸门验收 + 选择即验证路由" → 框架对 6 字段通用。

**可现场跑的 demo**（都在 `scripts/` 或可临时写）：

1. `route_field(REVENUE,'000425',2025)` → routed；`route_field(REVENUE,'300005',2025)` → needs_repair（**证明不张冠李戴**）。
2. 引擎跑 300005 → 成本/研发/员工 三字段路由命中、硬规则通过、带溯源。
3. 引擎跑 300009 → 客户/供应商各 5 行明细路由命中、占比和=前五合计。

## 8. 诚实的局限（被问"有什么不足"时答这个，显成熟）

1. **认表脆弱性**：研发费用表和销售/管理费用表同构，我靠"研发特征科目+勾稽消歧"认表，不是按"研发费用"小节标题锚定——per-layout 专用解析器的取舍，更稳的做法是章节锚定（已在准则文档里规划）。
2. **golden 规模小**：目前 3 份报告认证了 6 个解析器；要让指纹路由真正显威力需要规模化 golden + 解析器。
3. **沙箱隔离**：本进程 exec，批量跑前需子进程硬化。
4. **D 类判据弱**：客户/供应商准则上明细可缺失，自动判据弱，本质更依赖人审。
5. **规范驱动 prompt 不是银弹**：实测注入准则没能救活弱模型在"唯一行业=100%像总计"这种细微代码边界上的失败——靠 fork 给母本或换强模型解决，prompt 工程有上限。

## 10. 附录 A：指纹缓存路由深拆（`route_field` 第①步）

> 一句话：**指纹只负责"先试哪个"（快），硬规则永远负责"对不对"（准）。**

### A.1 它解决什么问题

"选择即验证"原本是：来一份报告，把该字段**所有候选解析器都跑一遍**，看谁过硬规则。目标是"一版式一解析器"，候选会涨到成百上千——每份全跑太慢。于是加缓存：**同版式的报告应该用同一个解析器**，第一次费劲选出来后记住，下次同版式直接命中。

### A.2 "指纹"是什么（`layout_fingerprint.compute_fingerprint`）

只读 PDF 文本、不解析表格（所以快），算一个代表"版式"的短哈希。抓两样：

- **doc_type**：银行/券商/保险/普通（按特征科目出现**次数**判，避免偶现误判）。
- **每类结构关键词首次出现的"相对页位置"**（分 10 档桶）：研发/员工/成本/客户/供应商/分产品…在文档前百分之几。

拼签名再 md5：

```
signature = "normal|client:2|cost:5|employee:6|rnd:5|..."
hash      = md5(signature)[:12]      # 如 "b4563c8f340f"
```

**关键**：同模板出的报告（不同公司/年份/页数略差），结构关键词的相对位置几乎一样 → 同一指纹。分桶+相对位置就是为了让"同版式不同年份"归到一起。

### A.3 两个缓存文件

- `fingerprint_cache.json`：`{"300005_2025": "b4563c8f340f"}` —— 指纹算过就存，不重算。
- `route_cache.json`：`{"cost_breakdown|b4563c8f340f": ".../cst_300005_v1.py"}` —— "**该字段+该指纹 → 上次选中哪个解析器**"。键是 `field|指纹`：同份报告营收/成本要用不同解析器，故按字段隔离。

### A.4 短路流程逐句拆

```python
fp = fingerprint_of(code, year)          # 版式指纹(缓存,O(1))
cached = route_get(field, fp)            # route_cache["field|fp"] → 上次选中的解析器
if cached and 文件还在:
    rb  = version_parse_fn(cached)(code, year)   # 只跑这一个
    sig = field_plausibility(spec, rb)           # ★ 重新过硬规则
    if sig.clean:
        return routed(cache_hit=True)            # 干净→直接用,候选一个没跑
    route_invalidate(field, fp)                  # 不干净→删缓存,回退全量选优
```

### A.5 题眼：缓存只加速，从不拍板对错

命中缓存后**没有盲信**，而是重新跑一遍、重新过硬规则：

- 干净 → 用它（省下跑 N 个候选的时间）。
- 不干净（指纹撞车/版式漂移/这份其实不一样）→ **作废缓存、回退全量选优**。

所以**指纹再不准，最坏只是"白命中一次再回退"，永远不会因缓存而出错**——这就是"**指纹精度不是承重点**"的含义：它只影响速度和召回，不影响正确性。

### A.6 具体追踪

**第一份**（300005 成本，冷启动 miss）：

```
fp=b456 → route_get 空 → 全量选优 → cst_300005_v1 过硬规则 → 选中
        → route_set: "cost_breakdown|b456" → cst_300005_v1   (下次直接命中)
        → tag_fingerprint: 给该解析器贴 b456 指纹             (自学,缩候选更准)
```

**第二份**（另一家公司，同模板同指纹 b456）：

```
fp=b456 → route_get 命中 → 跑 cst_300005_v1 → 过硬规则 → 干净 → 用
        （候选一个没跑,这就是加速）
```

### A.7 三级漏斗（缓存在全局的位置）

```
① 路由缓存命中  → 1 个解析器,过硬规则就用            ← 最快(route_get)
② 指纹缩候选    → 只跑"贴了该指纹"的少数候选+验证     ← candidates_for
③ 召回兜底      → 指纹没匹配 → 该字段全部候选都跑      ← 防漏对的(召回)
```

①②是优化、③保证不漏；每一级选出来的都要过硬规则才算数。

**面试一句话**：指纹是"版式的廉价代理"，路由缓存是"同版式记忆"，但记忆每次都被硬规则复核——**加速归加速，对错归硬规则**。
