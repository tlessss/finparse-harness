# FinParseAI 工程化提升计划书（2026 H2）

> 目标：从“可用的 Agent 功能系统”升级到“可运营的 Agent 平台系统”  
> 范围：围绕 `run_id`、事件观测、Eval 门禁、并发作业、SLO 体系分阶段落地  
> 周期：12 周（3 个阶段）

---

## 一、背景与问题定义

当前 FinParseAI 已具备：

- 运行期自愈链路：`diagnose -> L1/L2/L3 -> verify`
- 构建期复利链路：`auto_heal -> repair -> certify -> route_field`
- 批跑与结果：100 家全链自主入库率 81.3%
- 控制台与失败分类能力

现阶段主要短板：

1. **同案不同上下文**：`Pipeline / Debug / Failures` 不是统一 `run_id`
2. **可观测不完整**：有结果无完整过程，难以回放与复盘
3. **Eval 非门禁化**：有批跑结论，但缺 PR 级自动回归与阈值阻断
4. **作业系统弱**：并发、幂等、重试、死信、限流能力不足
5. **运营指标缺失**：缺 SLO 与告警体系

---

## 二、总目标（12 周）

### G1：运行全链可追踪

- 每次 `run_field(use_llm=True)` 具备唯一 `run_id`
- 所有关键步骤写入 `process_events`
- 任一 case 可一键时间线回放

### G2：改动可比较可阻断

- Prompt / Rule / Parser 变更自动触发回归
- 固定 golden set 输出可比较指标
- 未达阈值时阻断合并（门禁）

### G3：批跑可运营

- 并发作业具备队列、重试、死信、限流
- 形成稳定 SLO 看板与告警

---

## 三、分阶段实施计划

## Phase 1（第 1-4 周）：统一上下文 + 事件观测（P0）

### 目标

- 建立统一 `run_id` 与快照模型
- 打通事件采集、查询、展示

### 任务清单

1. **统一运行上下文**
   - 新增 `run_id` 生成与传递规范
   - 抽象 `case snapshot`（输入、配置、关键中间结果）
   - `Pipeline / Debug / Failures` 统一读取同一快照

2. **事件流标准化**
   - `process_events` 字段标准：`run_id, event_type, outcome, payload.input/output/summary`
   - 关键事件全覆盖：`input_ready/routed/cold_parse/anchor_check/verify/heal_select/rule_heal/extract_heal/diagnose/committed/human`
   - 约束 payload 大小与结构

3. **时间线 API 与回放**
   - `GET /pipeline/timeline` 支持按 `run_id` 查询
   - 补 `GET /pipeline/runs` 用于查看历史运行
   - 增加回放视图（先表格/列表，后图形）

### 验收标准

- 90% 以上 case 能返回完整时间线（>= 8 步）
- 任一失败 case 能在 3 分钟内复盘到“卡点步骤”
- Debug 与 Pipeline 对同一 `run_id` 输出一致

### 交付物

- `run_id` 规范文档
- `process_events` 数据字典
- 时间线 API 文档与前端最小视图

---

## Phase 2（第 5-8 周）：Eval 门禁化 + 配置追踪（P0/P1）

### 目标

- 把“跑批结论”升级为“可阻断门禁”
- 形成变更可追责链路

### 任务清单

1. **固定回归集与基线**
   - 确定 golden set（建议 100 家 + 难例池）
   - 固定指标：`committed_rate / verify_hold_rate / no_such_table_rate / latency_p50/p95`
   - 固定输出格式（JSON + markdown 报告）

2. **PR 自动回归**
   - 变更触发策略：Prompt/Rule/Parser/API
   - 自动跑回归并上传报告
   - 阈值门禁：未达标阻断合并

3. **配置指纹追踪**
   - 每次 run 记录 `prompt_version/rule_version/parser_version/model`
   - 引入 `config_fingerprint`
   - 支持“某次异常结果反查配置版本”

### 验收标准

- 100% PR 具备回归报告
- 关键指标有基线对比（当前 vs 基线）
- 变更可追责：任一结果能追溯到配置指纹

### 交付物

- Eval 门禁策略文档
- CI 回归脚本与模板报告
- 配置指纹规范与查询接口

---

## Phase 3（第 9-12 周）：并发作业 + SLO 运营（P1/P2）

### 目标

- 批跑具备平台化作业能力
- 建立稳定运营指标与告警

### 任务清单

1. **作业系统增强**
   - 任务状态机：`pending/running/retry/deadletter/success`
   - 幂等键：`(code, year, field, run_type, config_fingerprint)`
   - 重试策略：指数退避 + 最大重试次数
   - 死信重放机制

2. **限流与资源治理**
   - LLM 并发信号量
   - 按模型/接口限速
   - 成本与耗时统计

3. **SLO 与告警**
   - 建立 SLO：`verify_hold_rate`、`commit_rate`、`p95 latency`、`error_rate`
   - 告警策略：指标突增/突降、长时间卡死、死信异常增长
   - 周报机制：稳定性与质量趋势

### 验收标准

- 500+ case 并发批跑稳定无阻塞
- 死信有重放闭环，重放成功率可统计
- 每周自动产出 SLO 报告并可追踪趋势

### 交付物

- 作业系统设计文档
- SLO 指标定义与告警策略
- 周报模板与自动产出脚本

## 五、组织与分工建议

- **架构负责人（你）**：标准制定、关键设计评审、门禁阈值拍板
- **后端工程**：事件模型、作业系统、回归脚本
- **前端工程**：timeline 展示、复盘交互、指标看板
- **测试/数据**：黄金集维护、指标校验、回归报告审阅

---

## 六、风险与应对

1. **范围膨胀（边做边加功能）**
   - 应对：每阶段仅做目标清单，不插入“新大功能”

2. **事件模型不稳定导致前后端反复改**
   - 应对：先冻结 `payload` 协议，再做 UI

3. **回归耗时过长影响开发效率**
   - 应对：分层回归（快速集 + 全量夜跑）

4. **并发后稳定性下降**
   - 应对：先单机并发稳定，再扩展多机；优先幂等与死信机制

---

## 七、成功判定（Definition of Done）

满足以下 5 条即视为工程化升级成功：

1. 任一 case 有唯一 `run_id` 且可回放全链路
2. Prompt/Rule/Parser 变更都有自动回归报告
3. 关键指标未达阈值可自动阻断合并
4. 批跑具备重试、死信、限流、幂等
5. 有稳定 SLO 监控与周报机制
