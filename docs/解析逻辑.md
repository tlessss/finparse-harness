# FinParseAI PDF 解析逻辑

> 描述 `engine.run()` 把一份财报 PDF 解析成 6 个结构化字段的完整逻辑。
> 对应代码：`src/engine_orchestrator.py` + `src/parsers/`。
> 最后更新：2026-06-24

---

## 一、总流程

```
engine.run(pdf, stock_code, year)
        │
   ① scan_pdf(pdf)  ── 全文表格扫描（只扫一次，结果共享给所有解析器）
        │            pdfplumber 抽第 15~215 页的所有表格
        │            每张表打"章节标签"：附注 / 管理层讨论 / 其它
        │            （用 PyMuPDF 搜"财务报表附注"等关键词定位章节边界）
        │
   ② 为每个字段选解析器  selector.select_parser(field)
        │            扫 parsers/<字段>/ 目录下所有解析器类，
        │            逐个调 can_handle() 取置信度最高者
        │            （目前只有"营收"有 default/bank 两版会真正比分，其余各 1 个）
        │
   ③ 5 个解析器各自 parse(pdf, pre_scan=①的结果)
        │            营收 / 研发 / 员工 / 成本 / 供应商+客户
        │
   ④ 组装统一 JSON（6 字段 + parse_flags + field_count）
```

输出结构（节选）：
```json
{
  "revenue_breakdown": {"segments": [...], "industries": [...], "regions": [...]},
  "rnd_info": {"rnd_detail": [...], "total_this": ..., "total_last": ...},
  "employees": {"total": ..., "composition": [...], "education": [...]},
  "cost_breakdown": [...],
  "top_clients": {"items": [...], "total_ratio_pct": ...},
  "top_suppliers": {"items": [...]},
  "parse_flags": {...}, "field_count": 6
}
```

## 二、关键：解析器分成两套打法（不一致）

```
   ┌─ A. 共享扫描派 ───────────────┐    ┌─ B. 关键词翻页派 ──┐
   │  研发 / 员工 / 成本 / 供应商   │    │  营收(default+bank) │
   └───────────────────────────────┘    └────────────────────┘
```

### A. 共享扫描派（研发 / 员工 / 成本 / 供应商）
直接复用 ① 扫好的表，靠"特征签名"挑表：

```
filter_by_signature(tables, "rnd" | "employee" | "cost" | "supplier")
   对每张表打分（TABLE_SIGNATURES 配置）：
     + 在附注章节         +30
     + 命中 must_have 词   +25/个   (如研发要有"职工薪酬/研发材料")
     - 命中 exclude 词     -60      (如研发表里出现"营业收入"就排除)
     + 有占比列 / 金额列   +20 / +15
     - 行数不在区间        直接丢
     - 占比 > 100%         -50      (防止把"同比%"误当"占比%")
   → 取最高分的那 1 张表
        │
   detect_column_types(table)  ── 统计法认列（见第三节）
        │
   各解析器按自己的行逻辑抽取
     · 研发：逐行抽金额，"合计"行存为 total_this/total_last
     · 员工：遇"专业构成/教育程度"切换模式，逐行抽人数
     · 成本：第 1 列=行业，第 2 列=项目，扫整行找金额/占比
     · 供应商/客户：不挑表，遍历所有表，按文本匹配，split() 切词找序号/金额/占比
```

### B. 关键词翻页派（营收）
不用 ① 的共享扫描，自己重开 PDF 再扫一遍：

```
_find_candidate_pages()  PyMuPDF 搜"营业收入/分产品/分行业/分地区..."找候选页
        │
   重新用 pdfplumber 抽这些页的表
        │
   _filter_revenue_tables()  自己再打一套分（与 A 的签名逻辑重复但独立实现）
        │
   _classify()  按"分产品/分行业/分地区"标签切桶，逐行抽 name/amount/ratio
                · 跳合计/小计/总计行（is_total_row）
                · 跳表头行
                · 按名字去重
        │
   银行版(bank.py)：can_handle 命中"非利息净收入"≥2 次时接管，
                   按"利息净收入/非利息净收入"等科目抽取
```

## 三、两个横切公共工具

### 认列 `detect_column_types`（纯统计，不看表头）
对每一列统计有多少格"像文字 / 像钱 / 像百分比"，据此猜：
- **ratio_col**：该列出现多个 `%` 且数值落在 0~100
- **amount_col**：该列出现最多"像金额"的格子
- **name_col**：既不是金额也不是占比、且多为中文的列

> ⚠️ 完全不读表头文字（"本期/上期/占比"），靠数值分布猜，是脆弱点之一。

### 单位 `detect_unit`
正则搜"单位：人民币万元 / 货币单位：百万元"等 → 得到到「元」的倍率，
再用 `convert_to_yuan(value, ratio)` 把金额统一换算成元。

## 四、各字段一览

| 字段 | 打法 | 核心逻辑 | 解析器文件 |
|---|---|---|---|
| 营收 | B 翻页 | 找营收页 → 挑表 → 按分产品/行业/地区切桶逐行抽 | `revenue/default.py`、`revenue/bank.py` |
| 研发 | A 签名 | 找含"职工薪酬/研发材料"的表 → 认列 → 逐行抽，"合计"行存 total | `rnd/default.py` |
| 员工 | A 签名 | 找员工表 → 遇"专业构成/教育程度"切模式 → 逐行抽人数 | `employee/default.py` |
| 成本 | A 签名 | 找含"占营业成本比重"的表 → 第1列行业、第2列项目 → 扫行找金额/占比 | `cost/default.py` |
| 客户/供应商 | A（更土） | 不挑表，遍历所有表，按"客户名称+销售额"等文本匹配，split() 切词 | `top_supplier/default.py` |

## 五、为什么会漏行（当前主瓶颈的根因）

全量 1249 份跑批：硬规则清洁率 44%，死信 699 里 643 是营收 `ratio_sum`（占比之和偏离 100%），
而其中多数是"占比之和 57~87%"的**漏行**。根因：

1. **只取"最高分的 1 张表"**：若营收分散在多张表，或某些行被 pdfplumber 切到了另一张表，
   整段会丢失 → 占比之和 < 100%。
2. **认列靠统计、不靠表头**：有两个年度的占比列时易认错列；合并/多行单元格会让某些行抽不到数字 → 该行被跳过。
3. **行抽取依赖 pdfplumber 的格子切分**：跨页表、合并单元格、换行单元格一多就碎。
4. **硬编码关键词、不读规则**：换了措辞（如"按产品类别"）就匹配不到 → 整表丢失。
   （注：解析器目前完全不读 YAML `self.rule`，这也是优化 Agent 无法自修复的根因。）
5. **两套打法不统一**：营收自己重扫一遍（慢 + 与共享扫描的认列逻辑各写各的，维护割裂）。

## 六、一句话总结

现在是"**扫全表 → 按关键词/统计特征猜出哪张表、哪几列 → 逐行硬抽**"的纯启发式流水线。
最脆的两个环节是 **挑表（只取 1 张）** 和 **认列/认行（纯统计、不看表头）**——
一旦不准就漏行，这正是清洁率卡在 44% 的根本原因。

> 改进方向见 `docs/全自主解析计划表.md` 的 Phase 1.2（表头驱动列识别 + 多表/多行合并）
> 与 Phase 2.0（解析器 rule-driven 化）。

## 附录 A：真实追踪示例（可复现）

> 用 `scripts/trace_parse.py` 把营收解析的 6 步逐一打印，看真实 PDF 怎么走完全程。
> ```bash
> python3 -m scripts.trace_parse --pdf pdfs/多氟多-2025.pdf      # 成功案例
> python3 -m scripts.trace_parse --stock 000425 --year 2025      # 失败案例
> ```

### A.1 成功案例：多氟多 2025（营收）

| 步骤 | 结果 |
|---|---|
| ① 找页 | 关键词命中第 23~32 页 |
| ② 抽表 | pdfplumber 抽出 2 张含分产品/行业/地区的表 |
| ③ 挑表 | 选中 15 行的表 |
| ④ 认列 | 名称列=0、金额列=3、占比列=6（"占营业收入比重"那一列） |
| ⑤ 切桶抽行 | 分产品 5 项=100% ✅ / 分行业 2 项=100% ✅ / 分地区 2 项=100% ✅ |
| ⑥ 硬规则 | ✅ 通过 |

pdfplumber 还原的真实格子（前几行，∅=空）：
```
行0: [∅, ∅, ∅, ∅, '2025年', ∅,∅, ∅, ∅, ..., '2024年', ..., '同比增减']
行1: [∅, ∅, ∅, '金额', ∅,∅, '占营业收入比重', ...]
行2: [∅, '营业收入合计', ∅, '9,434,24...', ∅,∅,∅,∅, '100%', ...]   ← 合计行(被跳过)
行4: ['工业', ∅, ∅, '7,987,86...', ∅,∅, '84.67%', ...]            ← 数据行
```
一行被摊成 19 个格子、大半是空的；名称/金额/占比分散在第 0/3/6 列——靠**统计法**猜对了列，所以成功。

### A.2 失败案例：徐工机械 000425 2025（营收）

| 步骤 | 结果 |
|---|---|
| ① 找页 | 命中第 16~25 页 |
| ② 抽表 | 抽出 2 张表 |
| ③ 挑表 | 选中一张 10 行的表 —— **但这是"营收/成本/毛利率"表，不是"营收占比构成"表** |
| ④ 认列 | 占比列=3 —— **认成了"毛利率"列！** |
| ⑤ 切桶抽行 | 分产品=64.3% / 分行业=22.6% / 分地区=45.4% ⚠️（全 < 100%） |
| ⑥ 硬规则 | ❌ 红线拦下 |

格子（前几行）：
```
行0: [∅, '营业收入', '营业成本', '毛利率', '营收比上年', '成本比上年', '毛利率比上年']
行2: ['工程机械行业', '100,822...', '78,028...', '22.61%', ...]
行4: ['起重机械', '20,982...', '16,253...', '22.54%', ...]   ← 22.54% 是毛利率，不是营收占比
```

**根因（比"漏行"更精确）**：这张表里根本**没有"占营业收入比重"列**，统计法只看到一列 0~100 的百分数（毛利率）就当成了占比列；而且**一开始就挑错了表**（挑了毛利率表而非占比构成表）。于是把毛利率当营收占比，求和自然不到 100%。

> 这正好对应第五节的两个最脆环节：**挑表（只取 1 张、可能挑错）** 和 **认列（纯统计、把毛利率误当占比）**。
> 修复方向：④ 改成**表头驱动认列**（识别"占营业收入比重"表头，没有该列就不取占比 / 换表），③ 改成**带语义的挑表**（区分"占比构成表"与"毛利率表"）。
