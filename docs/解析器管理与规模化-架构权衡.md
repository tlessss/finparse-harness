# 解析器管理与规模化 — 架构权衡

> 背景：解析器多了之后怎么组织、怎么存、怎么避免"重试时全跑一遍"。
> 结论先行：**现在（6 个解析器）几乎什么都不用动**；本文记录"什么时候、因为什么信号、该改成什么"，避免过早优化，也避免到时拍脑袋。

---

## 0. 触发这次讨论的三个疑问
1. 所有解析器堆在 `versions/` 一个文件夹，重试是不是会把它们全试一遍？
2. 把它们分类是不是更好？
3. 代码以文件存在文件夹里好管理吗？要不要用 git 仓库 / 数据库？

---

## 1. 先纠正一个前提：路由**不扫文件夹**
route 和 repair 都不读 `versions/` 文件夹，读的是**认证清单 `goldset/certified_parsers.json`**，且进门第一刀就按 `field` 过滤。

代码链：`route_field` → `candidates_for(field, fp)` → `load_certified()`（读 manifest）→ 按 field 过滤 → 指纹缩候选。

推论：
- "重试会试所有解析器" → 实际只试**该字段的已认证解析器**（营收的报告不会去跑成本的）。
- 文件夹六字段混放**对路由零影响**——文件夹只是"文件住哪"，**索引是 manifest**。
- 一个 .py 躺在文件夹里但没 `certify()` 登记，**就不是候选**。

> 区分两个东西：`versions/` 文件夹 = 代码住处；`certified_parsers.json` = 谁是有效候选的登记表（`{key, path, field, fingerprints}`）。

---

## 2. 真正的成本：指纹 miss 时的"兜底扇出"
三级漏斗的第③级（召回兜底）：**指纹没匹配 → 跑"该字段全部已认证解析器"**。
- 营收认证了 800 个版式、新报告指纹没匹配 → 跑 800 次解析 + 800 次硬规则。这是最坏情况成本。

所以要优化的**不是文件夹**，是"指纹 miss 时的兜底扇出"。

---

## 3. 分类：两层，价值完全不同

| 分类方式 | 价值 | 对路由 |
|---|---|---|
| **A. 文件夹按字段/类型分子目录** | 只帮**人**好找好管 | 中性（路由不看文件夹）|
| **B. manifest 加"可靠粗桶"(doc_type/行业)再过滤** | **真正缩小兜底扇出** | 实质提速 |

**B 才有实质价值**。原则：
> **精度高的指纹不可靠（只配缩候选 + 回退）；粒度粗的 doc_type 可靠（可以硬过滤）。** 用可靠的粗桶砍扇出，是正解。

A（子文件夹）解决"人看着乱"，不解决性能；想做随手做，不急。

---

## 4. 存储：文件 vs git 仓库 vs 数据库

**先确立一个原则**：**"代码作为产物存在 git 里"是优点，不是缺点。** 我们的模式是"LLM 写代码"——每个认证解析器都是可 diff / 可 blame / 可 code review 的文件。搬进 DB 就丢了 diff/历史/grep/PR 审查，还得"从库取出再 exec"。**所以解析器代码建议一直留在 git。**

真正的痛点不是"文件多"，是下面两个，各有不同的迁移触发点：

| 痛点 | 现在(6个) | 触发迁移的信号 | 迁移到 |
|---|---|---|---|
| **扁平命名空间**（几百 .py 挤一目录，人难找） | 无所谓 | 几十个以上看着乱 | **子文件夹**(field/doc_type)，代码仍在 git |
| **manifest 是单个 json**（并发写冲突/合并难） | 无所谓 | **前端控制台多人/多进程并发认证** | 先拆**按字段多 manifest**，再上 **DB/注册表** |
| **元数据缺失**（谁认证/何时/对哪份 golden） | 能凑合 | 要审计/追溯 | manifest 加字段，或进 DB |

**关键洞察**：逼你换存储的**不是文件数量，是控制台开始并发认证**——那时单个 `certified_parsers.json` 会写争用。所以"manifest → DB"绑定的是**控制台上线**，不是文件数。即便索引进了 DB，也是 **"DB 存元数据/索引 + git 存代码"的混合模式**。

**独立 git 仓库**？只有当"解析器更新节奏 ≠ 主程序发版节奏、想独立部署"时才值。现在加 submodule 徒增复杂度，**不建议**。

---

## 5. doc_type 粗桶过滤：目的与时机

**目的**：给"指纹 miss 时的兜底全跑"设一个**可靠的上限**。

**机制**：新报告先判 doc_type（银行/券商/保险/普通，`compute_fingerprint` 已算，靠特征科目次数=硬事实）。兜底时只跑**同 doc_type** 的解析器：普通公司报告只跑普通类（不碰银行专用的），银行报告只跑银行类。

**为什么能"硬过滤"而指纹不能**：doc_type 可靠——普通公司的对解析器**一定**是普通类，按它过滤**永不排除对的**（召回零损失）；指纹会抖动，只能缩候选+回退、不能硬砍。**doc_type 是少数可以放心硬过滤的桶。**

**诚实的时机判断**：

| | 6 个（现在） | 几百个/字段（未来） |
|---|---|---|
| 兜底全跑 | 最多 6 次，省不了几微秒 | 几百次，doc_type 砍掉一大半 |
| 收益 | ≈ 0 | 显著 |

按"别为规模过早优化"的原则，**这件事现在也可以等**。它便宜（manifest 加 `doc_type` + `candidates_for` 多一行）、可靠、可逆，是"顺手买的保险"，但**不是现在的刚需**。
**触发点**：单字段认证解析器超过几十个、兜底肉眼可见地慢时再做。

---

## 6. 决策与待办

**现在就值得做（与规模无关）**
- [ ] manifest 条目补**元数据**：`certified_by` / `certified_at` / `golden_ref`（关乎可审计/可追溯，越早记越全）。

**等信号出现再做**
- [ ] **doc_type 粗桶过滤**（信号：单字段解析器 > 几十个，兜底变慢）。
- [ ] **子文件夹整理**（信号：一个目录文件多到人难找）。
- [ ] **manifest → DB/注册表**（信号：控制台多人/多进程并发认证，单 json 写争用）。

**明确先不做**
- 把解析器代码搬进 DB（丢 git diff/审查，得物化才能跑）。
- 独立 git 仓库 / submodule（无独立部署需求，徒增复杂）。
- 文件夹大改（路由本来不看文件夹）。

---

## 7. 一句话原则
**索引是 manifest，不是文件夹；代码留在 git 是优点；逼你升级存储的是"并发认证"而非"文件数量"；能硬过滤的只有可靠的粗桶(doc_type)，指纹只配缩候选 + 回退。**
