# 全自愈实现计划

> 配套 `docs/全自愈架构.md`(讲 **什么/为什么**);本文讲 **怎么做/什么顺序**——可执行的分阶段计划。
> 核心逻辑:**双锚兜底(金额+源文)让我们敢大胆自愈 → 分诊路由把失败派给对的 healer → 固化让它越跑越强 → 每步重跑量指标。**
> 目标:自主解析率 ~62% → 95%,正确率优先。

## 二、分阶段计划(按依赖排序)

每个阶段固定五问:**目标 / 落点 / 闸 / 回收哪类 / 工作量·收益·风险**。

### Phase 0 · 金额锚双口径(地基,最先做)

- **目标**:金额锚从 `Σ ≈ 营业收入` 改为 `Σ ≈ 营业收入 OR ≈ 主营业务收入`(各 ±3%)。**先摆正"什么算失败"**——把会计上合法的主营披露从失败里摘出来(依据见架构 6.2:格式准则第2号披露的是"主营业务分…",Σ = 主营 ≤ 营收)。
- **落点**:`src/eval/anchors.py`(取主营锚:附注"主营/其他/合计"表 或 `营收 − 其他业务收入`;取不到退单边容差)、`src/parsers/revenue_router.py::_attach_confidence`(接受任一口径)。
- **闸**:表身份仍由源文锚(复核)守,放宽口径坑表也进不来。
- **回收**:单维不齐 / 口径差里一批假失败;与选表 agent 的 `caliber_gap` 对齐(现在确定性金额锚比 LLM 选表还严)。
- **工作量** 小 · **收益** 高 · **风险** 低。

### Phase 1 · Tier 1 诊断路由器(骨架)

- **目标**:把 `_classify_failure` 从"分析器"升级成"路由器";`_nongreen_llm` 改为"按类派 healer";`judge_diagnose.next_action` 真正映射并执行 healer,而非一律交人工。
- **落点**:新建 `src/agents/heal_orchestrator.py`;重构 `src/pipeline.py::_nongreen_llm`。
- **回收**:现有 healer(选表自愈/L2/跨页/避坑)触达率立刻上来——**全空 8** 就有救(之前是没派对药)。
- **工作量** 中 · **收益** 高(所有后续都挂这骨架上)· **风险** 中(重构主链,需回归)。

### Phase 2 · 取锚自愈 + 金融股域外

- **目标**:无锚时从利润表抽营业收入当锚(给金额锚"复明");按名称/行业识别银行/券商/信托 → 标 `out_of_scope` 移出分母。
- **落点**:`src/eval/anchors.py`(取锚兜底)、`src/eval/domain_scope.py`、`src/pipeline.py::failure_analysis` 正名"金融域外"。
- **状态**:✅ 已落地 — `heal_anchors` / `derive_anchor_from_income_statement` / `is_financial_stock` / `out_of_scope` 终态 / 无锚路由分支。

### Phase 3 · 选表多轮 + L2 迭代 + 认列扩展

- **目标**:选表自愈单轮 → 扫 top-N 候选逐个过锚;L2 提 delta 不过锚 → 把"为何不过"回喂再提一版(最多 K 轮);`rule_heal.yaml` prompt 补"单位 override(万/千元)"与"名称别名/清洗"。
- **落点**:`src/agents/select_table_agent.py`、`src/agents/rule_heal_agent.py` + `src/prompts/templates/rule_heal.yaml`。
- **状态**:✅ 已落地 — `heal_select` 先 `sweep_topn`(向量8张过锚);`_heal_and_verify` 多轮 verify;`rule_heal` 最多2轮迭代+`unit_ratio_override`;`revenue/default` 读单位覆盖。

### Phase 4 · 抽表自愈(L3)+ 固化认证解析器

- **目标**:真漏行(pdfplumber 抽碎)→ 换 pdfplumber 参数重抽 / 渲染页面区域 LLM 抽行,过锚闸兜底;成功的自愈**固化成认证解析器**(经验库,下次 routed 免 LLM)。
- **落点**:新建 `src/agents/extract_heal_agent.py`;`src/parsers/infra/table_scanner.py` 暴露"单页可调参重抽";`consolidate()` 接 `parser_catalog` + `src/agents/code_generator.py`。
- **回收**:严重缺 / 中度缺 / 真漏行(少但最硬);**固化让覆盖复利增长**。
- **状态**:✅ 已落地 — `rescan_page`/`list_rescan_profiles`;`table_cache.patch_page`;`extract_heal` + `_extract_heal_and_verify` 接线;`consolidate_*` 固化;**`src/eval/extract_profiles.py` 读回**(`get_tables` 自动补丁缓存 + `fp_index` 同版式复用),L3 优先试已固化 profile → 表修好可直接 `routed`/绿灯,免再走选表/L2/L3 LLM 链。**待续**:视觉 LLM 抽行兜底。
- **工作量** 大 · **收益** 中(数量少但攻坚)+ 长期(经验库)· **风险** 中。

### Phase 5 · Tier 2 超级管家

- **目标**:先接**批量复盘驱动路线图**(离线、最安全、最快见价值);再接**链尽/歧义时的兜底编排**。
- **落点**:新建 `src/agents/steward_agent.py`。
- **前置**:得有几个 healer、失败尾巴变清晰,管家才有的可管、可复盘。
- **工作量** 中-高 · **收益** 治长尾 + 自驱路线图 · **风险** 控制触发频率(绝不 per-report)。

## 三、横切关注点(每阶段都做)

1. **回归护栏**:每个 Phase 落地后**重跑 100 家**,看 `/console/failures` 类别下降 + committed 上升 + **无 committed 掉出**(像改复核时那样先隔离归因、再确认)。
2. **双锚是安全底线**:金额锚 + 源文锚两个独立真值,healer 产物两个都过才入库——所以 Phase 2-4 能大胆堆 healer。
3. **固化 = 复利**:Phase 4 起成功自愈落认证解析器 → 跑得越多 routed 越多、越省 LLM。
4. **从 100 到全量**:现在 100 家验证;地基稳后用已有 ProcessPool 并发跑全量(~1247 已缓存 → ~5000 全 A 股),观测靠失败分析页。
5. **字段泛化**:现在聚焦营收;架构通用,healer 营收特化——成本/研发等后续按同框架复制。

## 四、依赖与顺序

```
Phase 0(摆正失败集)
   └→ Phase 1(骨架:按类派药)
         ├→ Phase 2(补无锚盲区)
         ├→ Phase 3(加深现有 healer)
         └→ Phase 4(攻真漏行 + 固化经验库)
               └→ Phase 5(管家:治长尾 + 自驱路线图)
```

- **0 必须最先**:不摆正"什么算失败",后面诊断/路由建在错的失败集上。
- **1 是骨架**:2/3/4 都是往路由器上挂 healer。
- **5 最后**:要有地基和清晰长尾,管家才有的可管。

## 五、里程碑(方向性,非承诺)

## 六、纪律

- 严格按 `0 → 1 → 2 …` 走,**每步一重跑一验证**,不跳步。
- 每步改动都问:**两个锚还都在守吗?**(金额口径是否合理、源文/复核是否看的是同一张表)。
- 拿数据说话:每个 Phase 用 `/console/failures` 的类别增减来证伪/证实,而不是凭感觉。
