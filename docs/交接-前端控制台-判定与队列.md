## 1. 分诊队列页（新增，最重要）

**它是什么**：所有"需要 LLM 写/改解析器"的待办清单。系统自动产出，人从这里派活、追踪。

**UI**：
- 顶部：汇总卡片（open 总数 + 按 reason 分桶计数 + 按字段分布）。
- 主体：可筛选表格，列 = `字段 | 报告(code/year) | reason | 置信度 | 状态 | 更新时间 | 操作`。
- 筛选：按 `reason`、`field`、`status`。
- 行操作：「审核」(跳审核台) / 「认领」(置 in_progress) / 「复核」(对 low_confidence 触发 LLM)。

**reason 枚举 + 含义 + 建议色**：
| reason | 含义 | 该干嘛 | 色 |
|---|---|---|---|
| 🟢 `ok` (status) | 路由 + **锚验证过**(分项和≈DB营收/成本/研发) | 可信，可放心 | 绿 |
| 🟡 `unverified` | 路由 + 过硬规则，但**无 DB 锚可验**(客户/供应商无锚；员工无金额锚) | 大概率对、未核验 → 抽查/跑 #2 | 黄 |
| 🟠 `low_confidence` | 有解析器但分项和**对不上** DB 权威值 | 可疑，复核 | 橙 |
| 🔴 `needs_write` | 没有解析器解得干净 | 写新解析器(给 golden→自愈) | 红 |
| `suspicious` | #2 LLM 裁判判可疑 | 改解析器 | 橙红 |
| `needs_human` | 无 golden/修复失败 | 纯人工 | 灰 |

> ⚠️**别给假安全感**：只有 🟢(锚验证过) 才算"可信"；🟡 是"解出来了但没外部核验"——客户/供应商/员工**天生无 DB 锚**，黄是它们的上限，要靠 **#2 LLM 裁判或人审**升级成可信。

**仪表盘要显示两个数字（这是操作者的"安全感"）**：
- **verified%（绿）**：已核验、真能信的占比 —— 核心指标(净通过率的故事)。
- **parsed%（绿+黄+橙）**：解析出数据的占比。
- 二者之间(黄+橙)= 解出来但待核验/可疑；红 = 缺口。

**status**：`open(待办) | ok(可信绿) | resolved(原问题已修,算可信)`。

**接口（已就绪 ✅）**：
```
GET /triage/summary
  → {"total":int, "verified":int,"verified_pct":float,   ← 绿,已核验
     "parsed":int,"parsed_pct":float,                     ← 绿+黄+橙,解出数据
     "open":int, "by_reason":{needs_write,unverified,low_confidence,...},
     "by_status":{ok,open,resolved}, "ok_by_field":{...}}

GET /triage/queue?status=open|ok|all&reason=&field=
  → {"records":[ <记录,见§5.1> ]}    status=ok 拉🟢可信清单；默认 open 拉待办

POST /triage/scan    body: {"codes":["000425",...],"year":2025}   分诊落台账
POST /triage/review  body: {"reason":"unverified","limit":20}     对黄/橙跑 #2:ok销账/suspicious改标
  → {"reviewed":[{code,year,field,verdict,summary,issues}]}
```

> 页面建议做成 Tab：**可信(绿)** / **待核验(黄)** / **要改(橙)** / **要写(红)**，顶部一行 verified%/parsed% 大数字。

## 2. 审核台（增强现有 `/review/task`）

**它是什么**：对单个(报告,字段)，看解析结果 + **PDF 原文红框对照** + 置信度 + LLM 裁判，人来定夺。

**现有 `/review/task?code=&year=` 已返回**（✅）：
- 解析结果各字段
- `溯源`：`{ "字段路径": {"page":int,"bbox":[x0,y0,x1,y1]} }`（用于在渲染页上画红框）
- 渲染页 base64 图
- `parser_code`：当前解析器源码

**要新增展示**：
- **置信度徽章**：每字段的 `confidence`(high/low/unknown) + `anchor`(DB权威值) vs 分项和。
- **LLM 裁判面板**：按需触发，展示 `verdict / issues(逐条:现值→正确值+错误类型+源文依据) / summary`。

**接口**：
```
GET  /review/task?code=&year=     ✅已有（建议后端补 confidence/anchor 字段，我来加）
POST /review/judge   body: {"code":"300005","year":2025,"field":"cost_breakdown"}   ⚠️待加
  → <verdict,见§5.3>   (跑 #2 溯源裁判;较慢~10-20s,前端给 loading)
```

**红框对照怎么画**：拿 `溯源[字段路径].bbox` + 渲染页图，按 bbox 在图上叠红框。bbox 坐标系与渲染页一致（PDF points，左上原点）。

## 5. 数据形状 reference

### 5.1 分诊记录（triage record）
```json
{
  "code": "300005", "year": 2025, "field": "cost_breakdown",
  "reason": "low_confidence",
  "signal": {"clean": true, "confidence": "low", "anchored": false, "diff_pct": null},
  "note": "", "status": "open",
  "created_at": "2026-06-26 10:00:00", "updated_at": "2026-06-26 10:00:00"
}
```

### 5.2 置信度信号（field_plausibility，进 /review/task 每字段）
```json
{"clean": true, "confidence": "high|low|unknown", "anchored": true, "anchor": 100822903512.12}
```
- `confidence=high` 绿、`low` 橙、`unknown` 灰（如客户/供应商无 DB 锚）。

### 5.3 LLM 裁判结果（/review/judge 返回）
```json
{
  "verdict": "ok | suspicious | unknown",
  "confidence": 0.0,
  "issues": [
    {"field":"segments[0].amount_yuan","current_value":7342029.88,
     "correct_value":73420298.81,"error_type":"unit_error","reason":"源文为73,420,298.81元，缩小10倍"}
  ],
  "summary": "一句话结论",
  "grounding": "溯源原表 | RAG检索",
  "field": "cost_breakdown"
}
```
- `error_type` 枚举：`unit_error`(单位错位) / `pnl_misid`(毛利率当占比/选错表) / `dim_leak`(维度串行) / `missing_row` / `extra_row` / `wrong_year` / `name_error` / `other`。
- `verdict=ok` 绿、`suspicious` 红；`issues` 逐条展示"现值 → 正确值 + 类型 + 源文依据"。

### 5.4 字段清单（6 个）
`revenue_breakdown`(营收) `cost_breakdown`(成本) `rnd_info`(研发) `employees`(员工) `top_clients`(前五客户) `top_suppliers`(前五供应商)
