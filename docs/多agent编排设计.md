# 多 Agent 编排设计（解析器注册表 + 人在回路认证）

> 目标：把 FinParseAI 从"一个共享解析器 + 假 agent（确定性函数）"重构为
> **多解析器实例（按版式）+ 真多智能体协作 + 人在回路认证**的自进化系统。
> 衡量：净通过率从 8.9% 持续爬升，且**正确率优先**（宁可置空，不可错填）。
> 创建：2026-06-24
> 配套：`docs/新版解析技术路线.md`（四组件解析管线）、`docs/自修复设计.md`、`docs/全自主解析计划表.md`

---

## 〇、一句话设计

来一份 PDF，先用**已认证的专用解析器**解（命中就免人工）；没有就**通用解析器冷启动**；还不行就让 **LLM 现写一个专用解析器**，跑通硬规则后交**人工对照原文 review**，批准即**认证入库**——下次同版式自动套用。**LLM 在构建期写代码，运行期跑的是冻结的确定性代码。**

---

## 一、设计观：三类节点，别把所有东西都做成 agent

判断一个节点该不该是 "agent"，只问一句：**它需要泛化判断吗？**

| 类型 | 是否用 LLM | 节点 | 为什么这样定 |
|---|---|---|---|
| 确定性工具 | ❌ | 抽表认列管线、硬规则校验、沙箱、注册表分发、写库 | 要可复现、可单测、零成本、永远一致 |
| LLM 智能体 | ✅ | 诊断、**生成专用解析器**、语义裁判、会话主管 | 措辞/版式千变万化，规则枚举不完，得"读懂再决定" |
| 编排 | — | 路由决策、黑板、迭代回边 | 只管状态流转，本身不"思考" |

> **题眼（面试可讲）**：我没把所有东西做成 agent——抽表认列是确定性的，上 LLM 反而丢可复现性；只有"读懂报错再泛化"的诊断和"写专用解析器"才值得做成 agent。**而且 agent 不在解析回路里，它在 CI 回路里**：产出的是被测试+人工门禁住的代码，不是每次重新幻觉的输出。

---

## 二、核心架构：解析器多实例化（按版式，不是一个共享库）

### 2.1 为什么多实例

"一个共享解析器让 LLM 改"的致命问题是**爆炸半径全局**：改 `revenue/default.py` 影响所有报告，必须跑全量回归才敢合并，重且慢。

**解法：解析器按版式多实例化。** 每个版式（极端情况每个 PDF）一个专用解析器，**爆炸半径 = 同版式集**。改它、生成它，炸不到别人。LLM 可以放开手写，最坏只影响一类报告。验证也从"别把别人改坏"简化成"**这一类对不对**"——正好是人工 review 能拍板的事。

### 2.2 注册表

```
解析器注册表 = { 版式key → 已认证专用解析器 }
```

- 单调增长：覆盖的版式越多，需要现写的越少。
- 即"经验库"的物理实现——经验不是存 YAML 补丁，而是**存解析器代码本身**。
- `通用解析器`（现 `parsers/<字段>/default.py`）保留为**冷启动**：它越强，冷启动命中越多，需生成的专用解析器越少。与 `新版解析技术路线.md` 的四组件管线互补（通用啃大众版式，专用啃长尾）。

---

## 三、路由：选择即验证（不靠猜，靠跑）

**核心洞察**：不需要**预测**哪个解析器合适，把候选都跑一遍，**硬规则直接告诉你哪个解对了**。这消灭了"猜错版式"那一整类错误。

```
新PDF → 抽表(贵，一次，所有解析器共用)
      → ① 【指纹缩候选】把注册表几百个解析器 → 缩到几个
              （便宜、只取文本；只要召回，不怕粗）
      → ② 【跑候选 + 硬规则】谁解出来干净就选谁
              （事后验证，接近真值的信号）
      → 都不干净 → ③ 触发【生成专用解析器】
```

两段各扬其长：

| 段 | 手段 | 角色 | 失败模式 | 兜底 |
|---|---|---|---|---|
| ① 缩候选 | 版式指纹（结构相似度，事前） | 召回导向的预筛 | 张冠李戴 | 由 ② 跑验证兜住 |
| ② 定选择 | 跑解析器 + 硬规则（结果，事后） | 精准的最终裁决 | 硬规则覆盖不到的凑数 | 由人工 review 兜住 |

**妙处**：指纹可以一直很粗——它只负责"别把对的那个排除掉"，对不对交给硬规则。**粗指纹 + 跑验证 = 又快又准。** 注册表还小时（几个解析器）可跳过指纹直接全跑，大了再加指纹缩候选。

> 版式指纹现状：`infra/layout_fingerprint.py` 只看"有哪些板块、大致在哪页"，**不看表内结构**。作为②的预筛（要召回）够用；若将来要它更准，可升级到感知表头/列序。**但本设计不强依赖指纹精度**——精度由跑验证保证。

## 五、关键机制：构建期改代码，运行期零 LLM

这是调和"让 LLM 改代码"与"正确率优先/可复现"的命门：

> **LLM 在「构建期」写/改专用解析器；「运行期」解析跑的是冻结的、确定性的、零成本的 Python。**

智能体不是"解析时调 LLM"，而是一个**"会写解析器的工程师智能体"**：离线读失败案例 → 写专用解析器 → 跑沙箱 → 人工认证 → 冻结进注册表。之后跑批解析时跑的是冻结代码，**和现在一样可复现、可单测、零 API 成本**。

破了天花板（专用解析器能重写认表认列、能修漏行），又没丢运行期确定性。

### 5.1 生成策略：fork 优先，从零写兜底

**默认不从零写，而是 fork 一个最像的已认证解析器来改。** 理由：大多数版式是少数母版的变体；从已跑通的代码改 10% 的差异，比对着空白写一个正确的复杂解析器错误率低一个量级（LLM 擅长"改这一点"，不擅长"从零写对"）；且继承了 base 已被认证的正确性，**验证面缩成 diff**，人工只审改动 → 认证更便宜 → 直接利好摊薄。

**致命前提：改副本 / 覆盖钩子，永不原地改 base。** 否则 base 服务的所有版式被波及 → 全局爆炸半径回来了，per-layout 隔离白做。两种安全机制：

| 机制 | 做法 | 爆炸半径 | 代价 |
|---|---|---|---|
| **复制再改（首选）** | 找最像的已认证解析器 → 拷一份 → 改副本 → 新文件，base 冻结 | =1（只这个变体）| 有重复代码 |
| 继承+钩子 | base 暴露可覆盖步骤（选表/认列/切行），变体 subclass 只 override 差异 | =1，但 base→变体**有耦合** | base 一改要回归所有变体 |

**先用"复制再改"**（零耦合、隔离完美）；等**同一种 override 在多个变体里反复出现**，再抽成"base+钩子"（抽基类重构，测试门把关）。"先允许重复、看清变化轴再抽象"——过早抽基类会坑。

**母本怎么选 + 谁决定 fork/从零**：天然落在 §三"选择即验证"里——跑候选时得出"哪个现有解析器最接近（红线最少/字段最多）"，那个就是 fork 母本。再由 LLM 判断：

```
最接近的解析器 + 它的结果 + 差在哪
  · 差一点点(认错一列/多一维度) → fork 改钩子   ← 多数，便宜
  · 结构全然不同               → 从零写        ← 少数真·新版式
```

结果是系统收敛成"**少数母版 + 一堆薄变体**"，比 N 个独立巨石好维护。经验库这时存 **patch/diff**，不是整个解析器。变体 override 多到接近重写时，**升级成独立解析器**（正常生命周期）。

## 六、统一解析器契约 + 溯源（人工 review 能对照原文的物理前提）

人工 review 要"看到解析结果**和** PDF 原文对照"，前提是**每个数字带着出处**。所以所有解析器（通用 + 专用）实现同一契约：

```python
def parse(tables, context) -> {
    "字段值": { "营收分项": [ {"name":"工程机械","收入":4.32e9,"占比":0.38}, ... ] },
    "溯源":   { "营收分项[0].收入": (页码=87, 表id=3, 单元格=(行2,列4)),
               "营收分项[0].占比": (页码=87, 表id=3, 单元格=(行2,列5)), ... }
}
```

**前置基建要求**：抽表层 `infra/table_scanner` 必须给每个表/单元格保留 **(页码, bbox 坐标)**，解析器取数时把坐标一路带到输出。没有溯源，人工 review 就是"在 300 页 PDF 里大海捞针"——不可用。

统一契约还让注册表能**动态分发**任意专用解析器（同一接口、即插即用）。

## 七、数据对错裁判（LLM 找茬 + 确定性放行）

裁判回答一个没有标准答案的问题：**凭什么说一份解析"对"？** 关键是一个不对称——

> **用 LLM 判"这里错了"算数；用 LLM 判"对、可以放行"不算数。** 抓错和背书是两件事，LLM 一强一弱。

| | LLM 强不强 | 能不能独断 |
|---|---|---|
| **发现错误**（漏行、认错列、选错表、量级离谱）| 强（读懂+对比是主场，高 recall）| ✅ 主力 |
| **背书"对、放行认证"** | 弱（同源背书 / 不校准 / 自信地错 / 无算术锚）| ❌ 不行 |

### 7.1 三件套（比"四层投票"干净）

```
确定性勾稽(便宜,常驻探针) ──┐
                          ├─► 裁决合成(分治,不投票)
LLM 裁判(对抗式找茬,主力) ─┤      · 勾稽红线 → 一票否决(确定性可信)
人工(首次认证 / 存疑兜底) ──┘      · LLM 找到错 → 触发重写 / 存疑
                                 · 谁都不能单凭 LLM"说对"就认证放行
```

- **确定性勾稽**（已有 `hard_rules.py`）：占比和≈100、三和相等、分项≤合计。不是用来判对，是当**零成本常驻探针**——已认证解析器跑新报告时靠它便宜探漂移，不必每批掏 LLM。管"算术一致"。
- **LLM 裁判**（吃掉旧设计的"对账+语义+向量"三层）：给它结果 + 原文（溯源圈定区域 + 整页兜底），**对抗式 prompt**——"假设这份是错的，挑出漏行/认错列/选错表"。管"漏填 / 选错表 / 认错列"——**正是头号死因，只能靠它**。
- **人工**：首次认证 + LLM 判存疑时兜底。

### 7.2 裁决合成 = 分治，不是投票

投票会出两种灾难：一票否决→弱信号假阳性淹没人工（摊薄崩）；多数通过→强信号被淹没（放行错数）。正确做法**按信号类型分治**：

| 信号 | 角色 | 权力 |
|---|---|---|
| 确定性勾稽红线 | 算术一致性 | **一票否决**（可信） |
| LLM 找到具体错误 | 漏行/认错列/选错表 | 触发重写 / 标存疑（**不独断放行**） |
| LLM"看着没问题" | —— | 仅供参考，**不构成认证依据** |

### 7.3 成本与复现（LLM 不在解析热路径）

- `temperature=0 + 按 (报告, 解析器版本) 缓存裁决`：同一对只判一次，之后读缓存 → **复现稳定、成本一次性**。
- 跑批重判用确定性勾稽探针兜漂移，**不必每批重新掏 LLM**。
- 所以 LLM 裁判在**认证环节**判一次冻一次，和"§五 运行期零 LLM"不冲突。

### 7.4 存疑分流阈值（调和 正确率 vs 摊薄）

"怀疑型裁判"（正确率优先 → 多疑 → 多存疑）与"少打扰人工"（摊薄）正面冲突。靠一个**可调阈值 + 分流**化解：存疑里高风险送人工、明确结构错直接走自动重写、低风险放过。**这个旋钮不定，裁判要么放水要么淹没人工。**

### 7.5 已知坑（诚实）

- **溯源自指**：解析器选错表 → 溯源指向错表 → 裁判被带偏（嫌疑人自证清白）。缓解：裁判**不只看溯源区域，给整页/候选表独立视野** + 对抗 prompt + 首次人工认证兜底。**未全解。**
- **漏行只能押 LLM**：脏表上 LLM 数行也会错；但确定性层够不着漏行，这是唯一解，认了。
- **同源背书**：写解析器与判对错若同模型同偏见，盲区相关。缓解：对抗立场 + 人工首认；必要时裁判换模型。

## 八、人工 review 认证关

### 8.1 界面（前端已有 React 地基）

```
┌─────────────────────────┬──────────────────────────────┐
│ 解析结果(结构化)          │  PDF 原文(渲染页 + 高亮)       │
│  分行业  收入    占比     │   ┌── 第87页 ──────────────┐  │
│ ▸工程机械 43.2亿 38% ◄────┼──►│ ...营业收入 4,320,..   │  │ ← 点结果行
│  备件     12.1亿 11%      │   │ [高亮原始单元格 bbox]   │  │   高亮原文
│  ...                     │   └────────────────────────┘  │
│ [✅批准认证][✏️改正][❌打回]│                              │
└─────────────────────────┴──────────────────────────────┘
```

点左边任意结果行 → 右边 PDF 跳到对应页、框出原始单元格 bbox。人眼一秒确认"43.2亿确实在这格"。

### 8.2 三种动作

| 动作 | 效果 |
|---|---|
| ✅ 批准认证 | 该专用解析器入注册表，**同版式后续 PDF 自动套用、免再 review** |
| ✏️ 改正 | 修好本份；修正作为**负反馈/few-shot 喂回生成智能体** |
| ❌ 打回 | 带人类反馈，让生成智能体重新写 |

### 8.3 摊薄：人力按"版式"付费，不按"报告"付费

**整个方案能跑得动的根本**：一个专用解析器被人批准认证后，所有同版式 PDF 免再 review。人工只在"新版式首现"时付一次费，摊薄到该版式所有报告。

→ 可行性完全取决于：**N 份 PDF 实际有多少种版式？版式数 ≪ 报告数 才赚。**（见 §11 命门）

## 九、认证、信任与漂移

- **认证即信任**：批准后同版式自动套用、免审。
- **裁判持续守门**（§七）：自动套用时仍跑确定性勾稽探针 + 按需 LLM 裁判。某份同版式 PDF 触红线 → 说明**版式漂移 / 指纹太粗** → 自动打回人工复审 / 触发重新生成。裁判是"信任"的持续保险丝。
- **版式换代**：公司明年换模板 → 结构变 → 新指纹 → 冷启动重新生成（正确行为，不会拿旧解析器硬套）。

## 十、安全闸 + 一条铁律

因为爆炸半径已被"按版式多实例"限制到同版式集，安全模型比"改共享库"轻，但仍需四道闸：

| 闸 | 作用 |
|---|---|
| ① 编辑面收窄 | 生成智能体只产出**单个专用解析器**（实现统一契约），不碰通用解析器、不满仓乱改 |
| ② 硬规则门 | 专用解析器产出必须过硬规则（占比和≈100、三和相等、分项≤合计）才进 review |
| ③ 人工认证门 | 对照原文人眼确认才入注册表——**最终真值闸** |
| ④ git 版本化 | 每个专用解析器是版本化文件，认证记录可追溯、可回滚 |

### 铁律：验证强度 = 自修复安全的上限

LLM 能写出"看着对、硬规则也绿、但实际错"的代码（凑数蒙混）。所以：
- **硬规则覆盖率**决定②能挡住多少（如新增硬规则："占比列必须由'占营业收入比重'类别名命中，不得来自毛利率"）。
- **人工 review** 是②挡不住时的最终闸（③）。
- 二者共同 bound 住自修复的可信度。**正确率优先：宁可置空/转人工，不可错填。**

## 十一、命门与诚实风险（不灌水）

| 命门 | 说明 | 对策 |
|---|---|---|
| **摊薄率** | 方案成立前提是"版式数 ≪ 报告数"。**M0 已测（1282 份）**：当前文档指纹 271 版式（含页位置噪声、过细，覆盖80%需65/95%需207、长尾165单例）；去桶后塌成 27 版式（980 挤一簇、过粗无区分力）。**真实需要的解析器数夹在 27～271 之间，文档级指纹测不准** | 文档指纹**不堪当认证 key**（已实锤）→ 复用单位走 §三"选择即验证" + §5.1 fork。**决定性数字**待"营收表头结构聚类"实验（见 §13 M0′）|
| **指纹粒度** | 太细→不摊薄；太粗→张冠李戴 | 本设计用"粗指纹缩候选 + 跑验证定选择"规避；指纹精度非强依赖 |
| **溯源工程量** | 抽表层要改造成保留页码+bbox，否则 review 不能对照原文 | 列为前置基建（§六） |
| **硬规则覆盖** | 是②的挡板上限，也是凑数蒙混的唯一自动防线 | 持续补硬规则；人工 review 兜底 |
| **生成质量/重试成本** | LLM 写的解析器可能跑不过沙箱 → 重试 K 次仍败 → 转纯人工 | few-shot 注入相似已认证解析器；负反馈回灌 |

### 11.1 角色清单

| Agent / 工具 | LLM? | 输入 | 输出 |
|---|---|---|---|

### 11.2 黑板（扩 `ParseState`）

```python
class ParseState(TypedDict, total=False):
    # 输入
    stock_code; report_year; pdf_path; ...
    # 抽表 + 指纹
    抽好的表: list        # 每表/单元格带 (页码, bbox)
    版式指纹: str
    候选解析器: list      # 缩候选结果
    # 跑验证
    各候选结果: list      # [(解析器id, 结果, 硬规则报告), ...]
    选中解析器: str
    解析结果: dict
    溯源: dict
    # 校验（§七 裁判）
    硬规则报告: dict
    裁判结果: dict        # {verdict:通过|存疑|红线, 找到的错[], 证据, 建议动作}
    # 生成 / 沙箱 / 认证
    解析留痕: dict        # 候选表、命中表头、被排除行、占比实测和 —— 喂诊断/生成
    生成的解析器代码: str
    沙箱结果: dict
    认证状态: str         # 待审 | 已认证 | 改正 | 打回
    # 决策
    route: str           # 归档 | 生成 | 人工review | 转人工
    iteration: int
```

`跑验证` 与 `生成` 节点都必须吐 `解析留痕`——这是诊断/生成"看得见内幕"的信息地基。

## 十三、里程碑与落地顺序

> 地基不牢，agent 越聪明越危险。严格按依赖顺序施工。

| 里程碑 | 内容 | 验收 |
|---|---|---|
| **M0 文档指纹测算** | ✅ 已做（`scripts/m0_fingerprint_survey.py`，1282 份）| 结论：文档指纹不堪当认证 key，真实解析器数夹在 27～271；通过率全面 ~3% |
| **M0′ 表头结构聚类（决定性）** | 抽营收表 → 按表头列序签名聚类，测"营收解析器"真实复用率 | 拿到落在 27～271 间的真数；判定每版式一解析器成立与否（代价≈一次 baseline 跑批）|
| **M1 溯源基建** | 抽表层保留 (页码,bbox)；统一解析器契约（带溯源） | 通用解析器输出可溯源到原始单元格 |
| **M2 注册表 + 路由** 🟡骨架完成 | 注册表分发 + 缩候选 + 跑验证 | ✅ `src/parsers/registry.py`(契约+GenericReportParser+选择即验证)+engine pre_scan，5/5单测(脏专用输给净通用)；待接批处理 |
| **M3 人工 review UI** | 结果↔原文对照 + 溯源高亮 + 三动作 | 人能对照原文一键认证 |
| **M4 代码沙箱** | 单份隔离跑 + 硬规则门 + git 版本化 | 候选解析器自动 accept/reject |
| **M5 生成专用解析器智能体** | LLM 写解析器 + few-shot + 负反馈回灌 | 死信生成→认证成功率 ≥50% |
| **M6 会话主管层** | 聊天框 + 工具调用循环 + 多轮记忆 | 自然语言驱动全流程 |

**依赖关系（必须点破）**：M5（生成智能体）依赖 M1（溯源/契约）+ M4（沙箱）。没有沙箱和契约，生成的解析器无从验证、无从分发。**先把验证地基和注册表搭好，最后才放生成智能体进去。**
