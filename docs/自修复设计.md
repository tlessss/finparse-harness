# FinParseAI 自修复设计

> 状态图例：✅ 已实现 / 🟡 部分 / ⬜ 待建 / ⚠️ 已知阻塞
> 依据：`docs/全自主解析计划表.md`（Phase 0/1/2/3）+ 代码实测（2026-06-23）
> 一句话目标：**发现错 → 自己诊断 → 自己改规则 → 沙箱验证 → 变好就留、变差就回滚 → 记成经验**，全程零人工（难例除外），正确率优先。

---

## 0. 电梯陈述（30 秒讲清）

FinParseAI 的解析是**纯规则**（pdfplumber 抽表 + YAML 规则，零 LLM）。自修复要解决的是"规则在某些版式上失效"。我们走**数据驱动的 YAML 规则自修复**：AI 只被允许改受限的规则旋钮（页码/列/关键词/排除项），不生成自由代码；每次修复都在**沙箱里重解析 + 用校验器当裁判**验证，变好才提交、变差自动回滚；成功的修复按**版式指纹**沉淀进经验库，同版式直接复用、跳过 LLM。LLM 是留给最难那步的弹药，不是主力。

### ⚠️ 两个必须先认清的阻塞
1. **解析器不读 YAML（总闸）**：`src/parsers/*` 的 `parse()` 不消费 `self.rule` → "改 YAML"对行为零影响 → 这就是 `optimizer.apply()` 空转的根因。**自修复的第 0 步是把解析器改成 rule-driven。**
2. **漏行不归自修复管**：当前 #1 死因是营收**漏抽产品/地区行**（死信 73/79 是 `revenue_breakdown:ratio_sum`）。这是**抽表/列识别**的正确性问题（计划表 Phase 1.2），YAML 旋钮调不出来。**自修复不要去碰漏行，否则在错的地方空转。**

## 2. 目标 / 非目标

**目标**
- 闭环自愈：诊断 → 改规则 → 沙箱验 → commit/rollback → 记经验，全自动。
- 正确率优先：宁可不修，绝不把对的改坏、绝不批量产错。
- 省 LLM：大部分修复走经验/启发式，只有真难的才调一次 LLM。

**非目标**
- 不追求 100% 自动：≤5% 难例归死信队列，允许人工。
- 本文不解决漏行（属 Phase 1.2 解析正确性，另开）。
- 不把"LLM 生成 Python 代码"作为主修复手段（见 §3）。

---

## 3. 核心决策：修复改什么？（YAML 规则）

| 路线 | 怎么修 | 取舍 | 采用 |
|---|---|---|---|
| **A. 改 YAML 规则** | AI 改受限旋钮（页码/列/关键词/排除项）→ 重解析 | 安全、可 diff、可人工 review、同版式可复用；受限于规则表达力 | ✅ **主路线** |
| B. 改 Python 补丁 | LLM 生成代码覆盖解析器方法 | 表达力强，但执行 LLM 代码：危险、难回滚、难复用 | ❌ 仅 YAML 实在表达不了时局部、且需人工签 |
| C. LLM 直接抽数据 | 放弃修规则，LLM 从原文抽 | 贵、易幻觉、不可复用 | 🟡 仅死信兜底，打低置信标记进人工 |

**决策理由（面试要点）**：目标是"正确率优先、绝不批量产错"。YAML 受限但**可控、可审计、可复用**；让 LLM 在一个**封闭的旋钮空间**里做选择题，而不是开放地写代码，是用约束换安全。

## 4. 架构：自修复闭环

```
                ┌─────────── 校验器（裁判）───────────┐
                │  hard_rules（勾稽红线）+ vector（BGE）│
                └───────────────┬─────────────────────┘
                                │ 异常报告(哪个字段/哪条红线)
                                ▼
  ┌──────────┐   ┌──────────────┐   ┌─────────────────┐   ┌──────────────┐
  │ 经验库   │──▶│  诊断器       │──▶│  沙箱验证        │──▶│  决策         │
  │ 命中?    │否 │ 产出"改哪个   │   │ 应用候选规则→    │   │ 变好→commit  │
  │ 命中→套  │   │  YAML 旋钮"   │   │ 重解析→重校验    │   │ 变差→rollback│
  └──────────┘   └──────────────┘   └─────────────────┘   └──────┬───────┘
                                                                  │成功
                                                                  ▼
                                                         ┌──────────────┐
                                                         │ 经验库沉淀   │
                                                         │ 按版式指纹   │
                                                         └──────────────┘
```

**组件职责**

| 组件 | 文件 | 职责 |
|---|---|---|
| 裁判 | `validators/hard_rules.py` + `vector_validator.py` | 判这份对没对、错在哪个字段哪条规则（自修复全靠它，**可信度要先证**，见 §7）|
| 经验库 | `experience_db.py` | 同版式已知修复直接套，跳过诊断（省 LLM）|
| 诊断器 | `agents/optimizer.py`（改造） | 输入异常报告 + 原文相关页 + 当前字段 YAML → 输出**结构化修复提案**（不是自由代码）|
| 沙箱 | `agents/sandbox.py`（✅ 已有框架） | 应用候选规则到副本 → 重解析这一份 → 重校验 → 给出前后分差 |
| 决策/编排 | `agents/iteration.py` | 接受/回滚、重试上限（≤3）、记经验 |

## 5. 分层修复策略（由轻到重，省 LLM）

```
① 经验命中：同版式指纹有已知修复 → 直接套用规则        0 LLM
② 确定性快修：规则内常见错（单位错、明显页码偏移）→ 启发式   0~低 LLM
③ LLM 诊断改 YAML：①② 没解决 → 1 次 LLM 出修复提案 → 沙箱验   1 LLM
④ 死信：③ 重试 ≤3 仍不过 → 归难例池（人工 / 可选 LLM 直抽兜底）
```

> 设计意图：让**大多数修复不花 LLM**，只有真正没见过的难例才上模型。这也是"能不用大模型就不用"哲学的延续。

## 6. 关键接口契约（开发依据）

### 6.1 解析器 rule-driven 契约（Phase 2.0 总闸）
解析器 `parse()` **必须**消费 `self.rule[field]` 的这些键（YAML 已有，只是没被读）：
```yaml
revenue_breakdown:
  table_keywords: ["营业收入构成", "主营业务分产品"]   # 定位页（替代硬编码 pages）
  pages: "24-26"                                       # fallback 默认页
  columns: { name: 0, current: 1, ratio: 3 }           # 列映射（替代写死索引）
  exclude_names: ["合计", "小计"]                       # 排除行（第一个可改的旋钮样板）
```
**验收**：改 YAML 的某个键，`parse()` 行为随之变化（现在不变 = 阻塞）。

### 6.2 修复提案 schema（诊断器输出，受限旋钮）
诊断器**只能**输出对白名单旋钮的修改，不能输出代码：
```json
{
  "field": "revenue_breakdown",
  "diagnosis": "占比列取错(取了金额列)",
  "edits": [
    {"key": "columns.ratio", "from": 2, "to": 3, "reason": "表头'占比%'在第4列"}
  ],
  "confidence": 0.8
}
```
> `edits[].key` 必须在该字段的**旋钮白名单**内，否则拒绝执行（安全边界）。

### 6.3 沙箱契约（已有框架，需对接）
```
sandbox.try_fix(pdf_path, field, rule_before, edits)
  → 1. rule_after = apply(rule_before, edits)   # 只在内存副本
    2. result_after = reparse(pdf_path, rule_after)
    3. score_after  = judge(result_after)
    4. return { accepted: score_after > score_before and 不破坏其他字段,
                score_before, score_after, rule_after }
```
**红线**：`accepted=False` 时一律 rollback，原 YAML 不动。

### 6.4 经验库 key（Phase 5）
```
key = layout_fingerprint(pdf)        # 当前只按 stock_code，要改成版式指纹
value = { field, edits, hit_count, last_ok }
```

## 7. 红线 / 安全（面试加分点）

1. **裁判先证可信**：自修复全靠校验器判"对没对"。先做金标准集（Phase 0.1，人工标 50~80 份），证明 `error_recall ≥ 0.95`（宁可误报不可漏报）。裁判不可信 → 自修复是空中楼阁。
2. **rollback 兜底**：任何修复都在沙箱里先验，变差立即回滚，**绝不把对的改坏**、绝不污染生产库。
3. **旋钮白名单**：AI 只能改预定义的安全旋钮，不能写自由代码、不能改解析逻辑结构。
4. **漏行隔离**：自修复不处理"漏抽行"（抽表正确性问题），交 Phase 1.2，避免在错的地方打转。

## 10. 风险 / 待定

- **裁判精度天花板**：向量/硬规则判不出的错（语义级分类错），自修复也救不了 → 需 §1 金标准量化。
- **旋钮表达力**：相当一部分错（尤其漏行）超出 YAML 旋钮能表达的范围 → 这部分走 Phase 1.2 / 人工，不强求自修复。
- **巨潮/数据源**：若改"按需下载"（见数据源讨论），自修复的"重解析"要能拿到同一份 PDF（临时缓存策略）。
- **并发安全**：沙箱改 YAML 副本，主 YAML 的 commit 要串行 / 加锁，避免批量并发时互相覆盖。
