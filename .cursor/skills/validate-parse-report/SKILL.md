# 解析结果校验与报告生成（含修复建议）

## 参数说明

### 参数1: 测试结果文件（必填）

JSON 测试结果文件路径，接受两种格式：

**格式A — 批量测试结果**（`batch_test_results_*.json`）：
```json
{"parser_version": "...", "results": [{"stock_code": "000001", ...}]}
```

**格式B — 单份结果**（管线输出的单份 JSON）：
```json
{"stock_code": "002407", "revenue_breakdown": {...}, ...}
```

### 参数2: 股票代码（可选）

- **不传**：对结果文件中所有股票逐份校验（批量模式）
- **传了**：只校验该股票代码对应的那一份（单份模式）

---

## 工作流

### Step 1: 定位测试数据

从参数中读取 JSON，过滤结果，找到每份的 PDF 缓存：
```bash
find ../book-agent/output/pdf_cache -name "{stock_code}_*" | head -3
```

### Step 2: 从 PDF 原文提取对照数据

用 PyMuPDF 读 PDF，对每个有数据的字段，搜索关键词找到原文中的标准数值。

| 字段 | 搜索关键词 |
|------|-----------|
| 营收结构 | "营业收入构成"、"主营业务分产品"、"分产品"、"分行业" |
| 研发费用 | "研发费用"（附注章节） |
| 员工数据 | "专业构成"、"教育程度"、"在职员工" |
| 成本构成 | "占营业成本比重"、"营业成本构成" |
| 前五大客户 | "前五名客户"、"主要销售客户" |
| 前五大供应商 | "前五名供应商"、"主要供应商" |

### Step 3: 逐项对比，记录差异类型

对每个字段，记录：

| 标记 | 含义 |
|------|------|
| ✅ | 名称和数值都匹配 |
| ❌ 单位 | 金额量级差 10/100/10000 倍 |
| ❌ 列识别 | 占比列识别错误（sum≠100% / ratio>100）|
| ❌ 误匹配 | 表类型识别错误（如银行资产负债表被当研发费用）|
| ➖ 缺失 | PDF 存在但解析没有 |
| ➕ 多余 | 解析多出 PDF 没有的项 |

**特殊规则**：
- 银行（000001 等）没有研发费用 → 解析出研发是误匹配
- PDF 原文中客户名称就是"第一名" → 算正确

### Step 4: 推导修复建议（新增）

对每个差异，根据下表推导具体的代码修改方案：

#### 差异 → 修复建议映射表

| 差异现象 | 根因定位 | 建议修改的代码 | 改什么 |
|---------|---------|--------------|--------|
| **金额差 1000000 倍**（如平安银行 88021 元应是 880 亿）| `unit_detector.py` 没检测到 PDF 中的单位标注 | `unit_detector.py` 的 `_UNIT_MAP` 或 `_UNIT_PATTERNS` | 增加单位模式或关键词，确保 PDF 页眉/页脚中的「人民币百万元」能被正则匹配到 |
| **金额差 10000 倍** | 解析器硬编码了 `÷10000` 但 PDF 单位就是元 | `revenue_parser.py` / `cost_parser.py` | 确认 `detect_unit` 返回了正确倍率，或手动兜底 |
| **占比和 ≠ 100%**（如 64%）| `_detect_columns` 选错了 `ratio_col` | `revenue_parser.py` 的 `_detect_columns` | 调整 `ratio_col` 选择逻辑：优先选 `%` 值完全在 1-100 且和接近 100 的列 |
| **单行占比 > 100** | 误匹配了 KPI 汇总表而非营收结构表 | `table_scanner.py` 的 `TABLE_SIGNATURES["revenue"]` 的 `exclude` | 加排除词：`"不良"`、`"覆盖率"`、`"充足率"`、`"净息差"` |
| **银行有研发费用** | 管理费用表含"职工薪酬"被误匹配 | `table_scanner.py` 的 `TABLE_SIGNATURES["rnd"]` 的 `exclude` | 加排除词：`"吸收存款"`、`"利息"`、`"发放贷款"` |
| **研发项数太多**（>15项）| 匹配到了非研发费用的其他附注表 | `table_scanner.py` 的 `TABLE_SIGNATURES["rnd"]` 的 `max_rows` 或 `must_have` | 缩小 `max_rows` 或加排除词 |
| **客户/供应商 items 为空但有汇总金额** | 汇总表先于明细表被匹配，明细表没有被赋值 | `top_supplier_parser.py` 的 `parse()` 中的匹配条件 | 明细表匹配条件改为 `result["top_clients"] is None or not result["top_clients"].get("items")` |
| **客户/供应商全空** | `scan_pdf` 扫描范围不够（默认 80 页）或关键词不匹配 | `table_scanner.py` 的 `scan_pdf` 的 `max_pages` | 增大 `max_pages`（如 200）确认有足够的扫描范围 |
| **员工教育程度为空** | 专业构成和教育程度被 pdfplumber 拆成两个独立表 | `employee_parser.py` 的 `parse()` | 合并所有匹配表的 `composition` 和 `education` 结果 |
| **金额数量级正确但占比全部自算** | `_detect_columns` 把同比增减列当成了占比列 | `revenue_parser.py` 的 `_detect_columns` | 在 ratio_col 选择中增加"和接近 100"的校验选项 |
| **同一股票多次测结果不一致** | 依赖了 `scan_pdf` 中表格遍历顺序 | `top_supplier_parser.py` / `table_scanner.py` | 明细表和汇总表的匹配条件不应依赖遍历顺序 |

#### 修复建议输出格式

在报告末尾的"修复建议"章节，每条建议的格式：

```markdown
### 问题：{现象描述}

**影响股票**：{stock_code}
**差异类型**：{单位错误 / 列识别 / 误匹配 / 缺失 / 多余}
**根因文件**：`{文件路径}`
**根因函数**：`{函数名}`
**建议修改**：
1. 在 `{位置}` 加/改 `{具体代码}`
2. 原因是：{一句话解释为什么要这么改}

**预期效果**：改了之后 {什么字段} 应该从 {当前值} 变成 {预期值}
```

### Step 5: 生成完整报告

#### 批量模式报告模板

报告保存到：`test_results/批量测试报告_{文件名}.md`

报告结构：
1. **总体统计** — 字段覆盖率、平均耗时
2. **逐份校验** — 对每份股票，逐字段对比（PDF 原文 vs 解析结果 + 对比标记）
3. **修复建议** — 对每个发现的问题，推导代码修改方案
4. **改进优先级** — 按影响范围排列修复建议（影响股票数最多的优先）

#### 单份模式报告模板

报告保存到：`test_results/{stock_code}_{year}_校验报告.md`

结构同上，只需 1 份股票 + 修复建议。

---

## 注意事项

1. 每份对比完成后输出进度：`[{i}/{total}] {stock_code} 完成`
2. 修复建议要具体到文件路径、函数名和修改方案
3. 不需要同时修复所有问题——按影响范围排优先级
